# S-1289 · The Epistemological Blind Spot — When Your Observability Stack Tells You What Happened but Not Why the Agent Chose It

Your agent ran for 45 minutes, called 23 tools, and produced a wrong answer. Your APM dashboard shows green for all 23 tool calls. Your latency was within SLO. But the answer is wrong. You open the trace and see a wall of tool calls with timestamps and return values — no internal reasoning, no hypothesis about why the agent diverged, no way to reconstruct the decision that led to the bad output. The observability stack measured the plumbing. The reasoning chain is invisible.

## Forces

- **Traditional APM assumes deterministic execution** — a microservice trace is a stack trace: cause precedes effect, the path is reproducible. An agent's reasoning chain is generated on the fly, probabilistic, and may differ every run even with identical inputs. Standard tracing captures the tool call, not the mental model that produced it.
- **Spans outlast the tools that created them** — by turn 15 of a long session, the LLM reasons over 40,000+ tokens of accumulated context. A tool called on turn 3 can cause a failure on turn 18. Standard traces with millisecond-to-second lifespans cannot capture this. (Zylos Research, April 2026 — https://zylos.ai/en/research/2026-04-29-agent-observability-production-debugging)
- **< 1 in 3 teams are satisfied with their observability solutions** — this was the finding from Cleanlab's survey of 1,837 enterprises with AI agents in production. The gap is not tooling availability; it is that existing tools were not designed for the reasoning-chains-need-reasoning-traces problem. (Cleanlab, 2025 — https://cleanlab.ai/ai-agents-in-production-2025)
- **63% of teams plan to improve observability and evaluation** — the largest planned investment area for production AI teams, ahead of model selection and tool development. The market recognizes the gap. (Cleanlab, 2025 — https://cleanlab.ai/ai-agents-in-production-2025)
- **Only 5% cite tool calling accuracy as their top challenge** — the hard problem is not knowing *whether* a tool was called; it is knowing *whether the right tool was called for the right reason*. Tool accuracy is table stakes. Reasoning traceability is the unsolved layer.

## The Move

Capture the reasoning chain as a first-class trace artifact — not just tool inputs/outputs, but the explicit hypothesis and context that surrounded each decision point.

- **Instrument at the reasoning level, not just the tool level.** Every LLM call produces a span with: the prompt fragment that was active, the retrieved memory context that was injected, the tool descriptions that were available, and the explicit reasoning that preceded the tool selection. This is the span that standard OTel auto-instrumentation misses — you have to emit it manually from the agent loop.
- **Use OTel GenAI semantic conventions as the schema.** The OpenTelemetry GenAI Working Group has ratified span attributes for model names, token counts, tool calls, and generation parameters. Align to these so specialized platforms (Langfuse, Arize Phoenix, LangSmith, AgentOps) can ingest your traces without custom parsing. (Zylos Research, April 2026 — https://zylos.ai/en/research/2026-04-04-ai-agent-observability-tracing-debugging)
- **Tag spans with semantic lineage.** A tool call span should carry: upstream tool that produced the data it operates on, session ID for cross-session continuity, policy/version stamp for replay, and a human-readable "what I was trying to accomplish" field populated by the agent at decision time. Without these, replaying a 30-step failure is archaeology.
- **Preserve failed traces as first-class artifacts.** The instinct is to discard errors; the value is in the failure traces. Store the full reasoning chain for every run that exceeded budget, produced a wrong answer, or triggered a guardrail. These become your eval dataset and your regression suite.
- **Build a "trace replay" debugging mode.** Zipkin and Jaeger assume deterministic execution for replay. Agent debugging requires replaying the *reasoning state* — the accumulated context at each step — not just the call sequence. Design your observability schema so a human can reconstruct the agent's mental state at step N, not just what tool it called.

## Gotchas

- **Auto-instrumentation captures plumbing, not decisions.** Most OTel agent integrations only capture LLM chat completions and tool call round-trips. The reasoning that led to the tool selection is inside the model's output — you must extract and emit it explicitly.
- **Long traces are expensive to store and slow to query.** A 45-minute agent session with 23 tool calls produces 50-100 spans. Naive storage of full reasoning chains per span hits cost limits fast. Budget for a tiered storage strategy: high-detail spans for failures, summary spans for successes.
- **The span schema will outlast your agent framework.** If you embed observability schema details in your agent code, framework changes will break historical comparability. Keep the schema in a separate instrumentation layer so you can migrate agents without losing trace lineage.
- **Without a session ID on every span, cross-session continuity is broken.** Multi-session agents (coding assistants, customer service bots) that span hours or days need session-scoped span tagging. Without it, you can never answer "did this agent encounter this problem before, and how did it resolve it?"
