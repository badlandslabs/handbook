# S-980 · The Agent Evaluation Stack: When Your Agent Looks Right But Isn't

Your agent passes every test. The demo works. The user reports are fine. Then a silent failure surfaces weeks later — wrong data returned, hallucinated citation, misexecuted refund. Traditional unit tests show green. The problem is that evaluating agents isn't software testing with deterministic outputs. It's measuring probabilistic reasoning chains where the same input can produce a correct answer through broken logic, or a wrong answer through reasonable-seeming steps. The move is a layered eval stack: trajectory metrics alongside outcome metrics, deterministic gates plus LLM-as-judge, and evaluation baked into CI, not appended to it.

## Forces

- **Task-completion benchmarks lie.** An agent that achieves 60% success on a single run can drop to 25% across eight runs (Galileo AI, "Agent Evaluation Framework," Jul 2026). Standard benchmarks measure outcomes but miss the reasoning path — and agents that get the right answer through terrible logic will eventually fail on edge cases.
- **Non-determinism breaks reproducibility.** The same input can produce different tool sequences, different intermediate reasoning, and different final outputs across runs. A single-pass evaluation gives you false confidence.
- **74% of production agents still rely on human-in-the-loop evaluation** rather than standardized benchmarks (Galileo AI citing arxiv:2512.04123, 2026). Manual review doesn't scale and doesn't catch regressions.
- **Eval quality compounds.** Teams that build evals early develop faster, catch regressions before they ship, and adopt new models with confidence. Teams that skip evals spend more time debugging production failures than building features (Anthropic, "Demystifying Evals for AI Agents," Jan 2026).
- **Agent scaffolding masks agent quality.** When evaluating, you're evaluating the model and the scaffold together. A less capable model paired with a better scaffold can outperform a more capable model with a poor scaffold (Cameron Wolfe, "Agent Evaluation: A Detailed Guide," 2025).

## The move

**Separate trajectory metrics from outcome metrics.** These are different signals. Outcome metrics ask "did the agent solve the task?" Trajectory metrics ask "did the agent solve it the right way?" An agent can return correct data through a hallucinated database query — the outcome looks fine, the trajectory is broken. Log and score both dimensions.

**Use three grader types in a tiered pipeline.** Code-based graders (deterministic assertions: `assert "refund" in output.lower()`) run fast and catch regressions on known patterns — use these first. LLM-as-judge handles open-ended quality (tone, relevance, groundedness) where no fixed assertion exists. Human evaluation stays for edge cases and calibration validation. Run code-based first, escalate ambiguous cases to LLM-as-judge, flag rare failures for human review (Anthropic, "Demystifying Evals," Jan 2026; DeepEval docs, 2025).

**Account for non-determinism with pass@k metrics.** Run each eval task k times. `pass@k` = at least one of k trials succeeds. `pass^k` = all k trials succeed (a consistency measure). For high-stakes agents, track both: high `pass@1` with low `pass^10` signals lucky behavior. Use `pass^k` as a consistency gate before production deployment (Machine Learning Mastery, "AI Agent Evaluation Roadmap," Jun 2026; InfoQ, "Evaluating AI Agents in Practice," Mar 2026).

**Evaluate the reasoning and action layers separately.** The reasoning layer (plan quality, plan adherence) answers "is the agent thinking correctly?" The action layer (tool correctness, argument correctness, execution order) answers "is the agent acting correctly?" DeepEval provides `PlanQualityMetric`, `PlanAdherenceMetric` for reasoning and `ToolCorrectnessMetric`, `ArgumentCorrectnessMetric` for action. Both must pass — a good plan with bad tool arguments still fails (Zylos Research, "AI Agent Testing & Evaluation Guide," Jan 2026).

**Build a 3-tier rubric with 7 core dimensions.** Top level: correctness (task completion), safety (injection resistance, PII leakage), efficiency (step count, token usage, latency). Sub-dimensions under each. Assign weight by risk profile — a medical agent weights safety at 60%, a research agent weights correctness at 70%. Tag every eval case to a dimension so you know which area regressed when scores drop (Galileo AI, "Agent Evaluation Framework," Jul 2026).

**Integrate evals into CI with three trigger mechanisms.** Commit-based triggers run the full suite on code changes. Schedule-based triggers detect model drift on production traffic. Event-driven triggers fire on deployment events or SLO breaches. A single eval run on every commit is the baseline; anything less leaves regressions undetected in production (Galileo AI, "Agent Evaluation Framework," Jul 2026).

**Calibrate LLM-as-judge against human labels before trusting scores.** Run judge outputs against a golden set of 20-50 human-graded examples. Target 0.80+ Spearman correlation. A judge with poor human alignment will optimize for the wrong thing at scale. Re-calibrate when you change models, not just when scores look wrong (Future AGI, "Best LLM Evaluation Frameworks 2026," Jan 2026; MLflow, "LLM-as-a-Judge," 2025).

## Gotchas

- **Don't evaluate only the final output.** The trajectory — tool call sequence, intermediate reasoning, plan adherence — is where most agents fail. A wrong answer with correct reasoning is a grader problem. A right answer with broken reasoning is a future failure waiting to happen.
- **Single-trial pass/fail is misleading.** Run each task k times (k≥3 for fast iteration, k≥10 for high-stakes deployments) and report pass@k and pass^k. A 70% pass@1 with 40% pass^10 tells you the agent is flaky even when the surface numbers look acceptable.
- **LLM-as-judge is a powerful signal but not ground truth.** Judges have their own biases, context window limits, and sensitivity to prompt phrasing. Always calibrate against human labels and never trust a judge you haven't validated on a golden set.
- **Eval sets go stale.** As your agent improves, your eval set must evolve. If your agent scores 95%+ consistently, the eval set has become too easy — add harder cases. Track pass rates over time: a plateau is a signal to refresh, not a signal you're done.
- **PII and safety evals are often skipped until a breach.** Inject PII detection and prompt injection tests into your eval suite from day one. These failure modes are expensive to recover from and cheap to catch proactively.
