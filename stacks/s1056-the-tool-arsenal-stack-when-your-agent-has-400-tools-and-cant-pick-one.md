# S-1056 · The Tool Arsenal Stack — When Your Agent Has 400 Tools and Can't Pick One

You gave your agent every tool it could ever need. Search, file system, GitHub, Slack, Notion, Stripe, Postgres, Redis, browser, code execution, Maps, and the Hacker News MCP. It has everything. It can't decide what to use. It calls the wrong tool, loops, and produces garbage. The problem isn't that it lacks tools — it's that you gave it too many. The real discipline in agent tool design is knowing what to withhold.

## Forces

- **More tools = more decision entropy.** Every additional tool the model sees in its context expands the space of wrong choices. The agent must reason over which tool to invoke and how to interpret its output — compounding errors at each step.
- **Tool descriptions are a contract that decays.** APIs evolve. The model matches its behavior to the description it was trained on. A tool that worked perfectly six months ago silently degrades as the underlying API changes and the description doesn't.
- **Security surface grows with every tool.** Each MCP server that touches a production API — Stripe, Salesforce, internal DBs — is a potential credential exposure vector if the agent's context is logged, stored, or transmitted carelessly.
- **The demo environment vs. the production environment gap.** Tools that seem useful in a demo (access to everything) are often harmful in production (too many failure modes to reason about).

## The move

**Start with the minimum viable toolset — add only when removal would make the agent worse.**

- **Lead with documentation retrieval.** The single highest-leverage tool for coding agents is an up-to-date library docs MCP server (e.g., Context7). Before any API integration, before any file system access: does the agent know what the libraries it will touch actually do? Fix that gap first.
- **Add tools that remove steps you'd otherwise do manually.** If you still wouldn't reach for a tool yourself to do the task, the agent probably doesn't need it. The practical filter from Context7's guidance: *"What do you keep pasting into chat before the agent gives you a useful answer? If a server replaces that step, it belongs."*
- **Scope credentials to the task, not the session.** Never embed API keys in prompts, config files, or shared context. Use short-lived, scope-limited tokens per tool that expire after the task completes. This is now one of the highest-risk parts of production agent deployments (Guardrly, APIsec University, Stacklok all flag this independently).
- **Use MCP for consumer-facing integrations, direct API calls for backend.** Per HN discussion on the MCP spec: *"Once I realized it's actually more consumer-facing than the backend, it made much more sense as why it became so popular."* Backend monoliths don't need 500 MCP servers — they need clean API modules.
- **Browser automation > scattered API integrations for web research.** Browser-use + Gemini, Playwright-based MCP agents, and Agent-Browser (Vercel) are the practical consensus for agents that need to interact with the web. One browser tool covers all web destinations; N dedicated API integrations cover N destinations and require N credential sets.
- **Human-in-the-loop for sensitive writes.** Anthropic's production guidance explicitly calls out issuing refunds, updating tickets, and similar destructive actions as requiring human confirmation. Usage-based pricing companies charge only for successful resolutions — the confidence comes from keeping humans in the loop on irreversible actions.
- **Enforce a tool review process.** Before adding a new MCP server to production: (1) Would removing it make the next task measurably worse? (2) Does it introduce new credential scope? (3) Can you write an eval that confirms it's called correctly? If no to any, don't add it.

## Gotchas

- **Adding a tool because it exists.** The existence of an MCP server is not a reason to use it. The test is whether removing it makes the next agent task noticeably worse.
- **Embedding credentials in JSON manifests or prompts.** This is flagged as an anti-pattern by every security source reviewed. If a credential ends up in a logged context window, it's effectively exposed. Use scoped, short-lived tokens.
- **Tool description drift.** When the underlying API changes, the tool description the model was trained on diverges. Build a process to keep tool schemas in sync with API reality — stale descriptions are silent failures.
- **Overestimating what a browser tool can replace.** Browser automation works for research and simple task completion. It cannot replace deep API integrations for transactional or rate-limited operations.
- **Giving every agent the same toolset.** A coding agent, a research agent, and a data pipeline agent have different tool needs. One-size-fits-all tool provisioning increases entropy across all three.
