# S-1558 · The Multi-Agent Reliability Divide Stack — When Adding Agents Makes Your System Less Reliable

You added a researcher agent, a writer agent, a reviewer agent, and a coordinator. Context overflow is gone. Latency dropped. But your end-to-end success rate fell from 94% to 81% — and you didn't notice until a client complained. Multi-agent architectures promise specialization and parallelism. In practice, each additional agent multiplies your failure surface, and the failures are silent: confident answers built on broken sub-tasks, not exceptions. The divide between demo and production is not the code. It is the reliability math.

## Forces

- **Reliability compounds against you.** Five agents each at 95% reliability deliver ~77% end-to-end success. More agents is not more robustness — it is more probabilistic handoffs.
- **Coordination overhead devours budget.** Coordination overhead starts eating LLM budget once the task graph passes 6 agents. Beyond that, you are paying for the agent infrastructure, not the task output.
- **Failure modes are silent, not loud.** Multi-agent systems rarely throw clean exceptions. They return confident, plausible answers built on broken sub-task outputs. The output looks fine; the reasoning chain is severed.
- **Context explosion at handoffs.** Passing the full conversation thread between agents — as some practitioners recommend — causes context to inflate with each hop. By agent four, your downstream reasoning quality has dropped sharply.
- **Trust boundaries are load-bearing, not decorative.** The orchestrator-worker topology with an explicit trust boundary between coordinator and isolated worker agent containers has become the dominant production pattern — not for organizational reasons, but because unvalidated inter-agent outputs will corrupt downstream results.

## The move

- **Treat multi-agent as a last resort, not a first design.** Start with a single agent + tools. Decompose into multiple agents only when distinct, separable capabilities genuinely justify the added coordination cost. Princeton NLP found single agents match or outperform multi-agent systems on 64% of benchmarked tasks, adding 2.1 percentage points of accuracy at roughly 2× the cost.
- **Default to orchestrator-worker topology with explicit trust boundaries.** The coordinator owns the plan and final synthesis; workers execute independently. Never let a worker agent output flow directly to the next agent without validation at the boundary. The trust boundary is infrastructure, not organizational policy.
- **Pin agents to narrow, stateless roles.** Each agent should do one thing: researcher, reviewer, synthesizer. Stateless agents — no local conversation history, all state in a central store — enable independent scaling and horizontal pod autoscaling without context drift.
- **Replace direct agent-to-agent calls with an async message bus (NATS, Redis Pub/Sub, or Kafka).** Synchronous HTTP calls between agents create cascading failures and limit throughput. The orchestrator pushes state to Redis or Postgres; workers subscribe to topics.
- **Add critic/verifier steps at every high-stakes handoff.** Before an agent output flows to the next agent, a lightweight validation step checks structured output correctness, completeness, and schema compliance. Do not let a bad sub-output compound silently.
- **Hard-cap loops and iterations.** Set `max_iterations=15` as a floor. Without bounds, a cycle of agents handing off to each other can burn a month's token budget in minutes. Add circuit breakers and dead-letter queues so a failed agent triggers a fallback, not a retry loop.
- **Treat conversation threads as the unit of context, not the latest message.** When context must flow across agents, pass the full thread — every prior message, every research result. Truncating context at each hop destroys reasoning quality downstream.
- **Centralize observability at the orchestrator.** Each agent run should emit structured traces (OpenTelemetry) to a central store. The coordinator should surface a unified trace graph showing which agent ran, what it received, what it returned, and where latency lives.

## Gotchas

- **Adding agents makes things less reliable by default.** The naive assumption that 3 agents × 95% reliability = better output is wrong. You get ~86% end-to-end reliability. You need explicit design — critic steps, trust boundaries, circuit breakers — to make multi-agent better than single-agent.
- **Context inflation at each handoff.** Passing the full conversation thread between agents causes context to grow linearly with the number of hops. Use structured summaries or state stores instead of raw message passing for long chains.
- **The 6-agent ceiling.** In CrewAI benchmarks, coordination overhead starts consuming LLM budget when the task graph exceeds 6 agents. The fix is not more capable agents — it is a flatter topology or a split into independent sub-graphs.
- **Silent failures dominate.** Multi-agent systems almost never surface the fact that a sub-task failed. The coordinator receives a plausible output, acts on it, and propagates the error. Every agent boundary needs a non-negotiable validation step.
- **Cost runaway is a first-class risk.** Without hard iteration caps and circuit breakers, a circular handoff between agents can exhaust a token budget in minutes. Treat this as an infrastructure concern, not a prompt engineering concern.
