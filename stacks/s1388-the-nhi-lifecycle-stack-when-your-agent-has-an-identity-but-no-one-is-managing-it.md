# S-1388 · The NHI Lifecycle Stack — When Your Agent Has an Identity But No One Is Managing It

Your AI agent sends invoices, reads your CRM, triggers infrastructure deployments, and accesses customer PII. It has credentials. It has OAuth scopes. It has an IAM role. Somewhere in your IdP, there is a service account tied to it. Nobody created that account in the normal provisioning workflow. Nobody assigned an owner. Nobody scheduled a rotation. Nobody will clean it up when the agent is decommissioned. That account — and every secret attached to it — is now a Non-Human Identity (NHI), and it is your largest unmanaged attack surface.

## Forces

- **Agents self-provision identities.** Unlike human employees who go through IT onboarding, agents are created by engineers who need them to work. The fastest path is a service account + API key + broad permissions. This bypasses every identity governance process your security team built for human employees.
- **NHI population outpaces human identity by orders of magnitude.** Palo Alto Networks found that machines and agents outnumber human employees 82-to-1 at enterprise scale. A Cloud Security Alliance (CSA) and Oasis Security survey found most companies have no formal policies for creating or removing AI identities, and very few treat agents as independent identity-bearing entities.
- **Agent NHIs accumulate silently.** Every agent deployment adds credentials, API keys, OAuth tokens, and scoped permissions. Unlike human offboarding (calendar reminder, IT ticket, manager sign-off), there is no equivalent decommissioning ceremony. Agents get replaced, refactored, or abandoned — and their credentials stay live.
- **Compromised agents are indistinguishable from authorized ones.** Wendi Whitmore, Chief Security Intelligence Officer at Palo Alto Networks, called AI agents the single biggest insider threat of 2026. A compromised agent carries valid credentials, passes auth checks, and triggers actions within its permission scope. Your SIEM sees a legitimate actor.

## The Move

The NHI lifecycle for agents has five phases, each requiring explicit ownership:

### 1. Provisioning — Birth with a Purpose

Every agent identity gets created through a deliberate workflow, not a developer's terminal:

```
# NHI Provisioning Policy (pseudocode)
AGENT_ID = agent_registry.register(
    owner=engineering_team,
    purpose="invoice_processing",
    data_classification="PII",
    access_boundaries=["read_orders", "send_emails"],
    ttl_days=90,
    review_date=date.today() + timedelta(days=60)
)
CREDENTIALS = secrets_manager.create_scoped_key(
    principal=AGENT_ID,
    permissions=["orders:read", "email:send"],
    time_limit_days=90,
    ip_allowlist=[agent_infra_range]
)
```

Each agent gets: a unique registry entry, a purpose tag, a data classification, explicit permission boundaries (not broad IAM roles), a TTL, and an owner. The credential is issued by a secrets manager, not generated by the agent code.

### 2. Least-Privilege Binding — Narrow the Blast Radius

Agents must operate with the minimum permission set required for their task. This is structurally different from human IAM because agents are programmatic — permissions can be expressed as precise action-level scopes rather than role-level access:

```
# Permission boundary for an invoice-processing agent
ALLOWED_ACTIONS = {
    "read": ["orders:get", "customers:read"],
    "write": ["invoices:create:draft"],
    "never": ["orders:delete", "customers:delete", "payments:write"]
}

# Policy-enforcing proxy sits in front of every downstream API
def enforce_permission_boundary(tool_call, agent_nhi):
    if tool_call.action not in ALLOWED_ACTIONS.get(agent_nhi.data_classification, {}):
        audit_log.block(
            agent=agent_nhi.id,
            action=tool_call.action,
            reason="outside_permission_boundary"
        )
        return BLOCK
    return ALLOW
```

The policy enforcement layer must sit at the API gateway, not in the agent's code. The agent cannot be its own gatekeeper.

### 3. Continuous Inventory — Know What You Have

Agent NHIs are discovered and tracked in a central registry:

```
# Agent NHI Registry schema
class AgentNHI:
    id: str                    # Unique registry ID
    name: str                  # Human-readable name
    owner: str                 # Engineering team
    purpose: str               # Business function
    credential_type: str       # api_key | oauth | iam_role | mcp_token
    last_rotated: date
    ttl_expires: date
    last_activity: datetime
    permission_scope: list[str]
    data_classification: str   # PUBLIC | INTERNAL | CONFIDENTIAL | PII
    status: str                # ACTIVE | SUSPENDED | DECOMMISSIONED
```

Run continuous discovery: scan cloud IAM, secrets managers, IdP applications, and MCP server registrations for agent-like identities. Flag orphaned credentials (no recent activity, no owner, expired TTL) for review.

### 4. Rotation — Secrets Have an Expiration Date

Every agent credential rotates on a schedule tied to its sensitivity classification:

```
# Credential rotation policy
ROTATION_SCHEDULE = {
    "PII": 30,      # 30-day rotation for agents handling PII
    "CONFIDENTIAL": 60,
    "INTERNAL": 90,
    "PUBLIC": 180,
}

def rotate_if_expired(agent_nhi):
    days_since_rotation = (today - agent_nhi.last_rotated).days
    threshold = ROTATION_SCHEDULE[agent_nhi.data_classification]
    if days_since_rotation >= threshold:
        trigger_rotation_workflow(agent_nhi)
```

Agents must survive credential rotation gracefully — this means fetching fresh credentials from the secrets manager at runtime rather than embedding long-lived secrets in code.

### 5. Decommissioning — The Offboarding Ceremony

When an agent is retired, three things must happen atomically:

1. **Revoke all credentials** — invalidate every API key, OAuth token, and scoped permission associated with the agent's NHI.
2. **Audit the blast radius** — before revocation, enumerate every resource the agent accessed in the last 90 days. If any access was to sensitive data, trigger a security review.
3. **Remove from registry** — mark the NHI as `DECOMMISSIONED`, archive the audit trail, and notify the owner.

```
# Decommission checklist
def decommission_agent(agent_id):
    credentials = registry.find_credentials(owner=agent_id)
    for cred in credentials:
        secrets_manager.revoke(cred.id)
        audit_log.record(
            action="REVOKED",
            principal=agent_id,
            credential=cred.id,
            revoked_by=current_user,
            timestamp=now()
        )
    # Run blast-radius audit before finalizing
    access_audit = audit_log.query(
        principal=agent_id,
        window_days=90,
        resource_type=["PII", "PAYMENT", "INFRASTRUCTURE"]
    )
    if access_audit.has_sensitive_access():
        security_review.trigger(agent_id, access_audit)
    registry.decommission(agent_id)
```

## The Governance Gap Behind the NHI Gap

The 93% governance gap (93% of organizations use AI, only 7% have fully embedded governance) is largely an NHI management failure. Agents proliferate faster than any human-run provisioning workflow can handle. The fix is not faster human approvals — it is a machine-readable NHI governance layer that enforces the provisioning, scoping, rotation, and decommissioning lifecycle automatically.

Treat every agent the way you treat a critical infrastructure service account: full lifecycle ownership, least-privilege scoping, automated rotation, and a formal decommissioning ceremony. The difference is that agents change faster, proliferate more silently, and carry broader implied permissions than any traditional service account.
