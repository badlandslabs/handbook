# S-1445 · The Agent Evaluation Stack — When Your Agent Looks Perfect and You Have No Idea If That's True

You shipped your agent. It passes every test. Your golden dataset returns green. And then a production user asks it to do the exact same thing three ways — and it works once, hallucinates twice, and times out once. Evaluation is the discipline of knowing whether your agent actually works before your users find out it doesn't.

## Forces

- **The silent failure problem.** An agent can produce a correct output through an incorrect process. Google Cloud's evaluation guide calls this out explicitly: an agent tasked with reporting inventory might give the correct number but reference last year's report by mistake. The result looks right. The execution failed. Traditional right/wrong metrics provide no diagnostic signal about where the breakdown occurred. — [Google Cloud Blog, Nov 2025](https://cloud.google.com/blog/topics/developers-practitioners/a-methodical-approach-to-agent-evaluation)
- **Infinite input space vs. finite test coverage.** Traditional software has bounded inputs. Agents accept unbounded natural language. The same user intent can be expressed hundreds of ways. You cannot enumerate all paths, so you must evaluate across distributions, not just test cases. — [LangChain Production Monitoring Blog, Feb 2026](https://www.langchain.com/blog/production-monitoring)
- **Trajectories, not just outcomes.** The sequence of tool calls and reasoning steps matters as much as the final answer. A working output from a broken trajectory is a ticking time bomb — it works today because of a lucky coincidence, and breaks on the next prompt variation.
- **Regression risk scales with iteration speed.** Agents often break when you tweak system prompts, add tools, or swap models. Teams shipping fast need evaluation to gate every deploy automatically — not as a manual step. — [Zalor HN Launch, 2025](https://news.ycombinator.com/item?id=47270208)

## The Move

The evaluation stack has four layers that work together:

- **Layer 1 — Define success criteria before building.** Name the specific behaviors you care about (task completion, tool call correctness, handoff quality, cost efficiency, safety) before you write the first prompt. Evaluation-driven development means building the eval harness before the agent. — [Confident AI Guide, 2025](https://www.confident-ai.com/blog/definitive-ai-agent-evaluation-guide)
- **Layer 2 — Build the data foundation.** Golden datasets (curated, annotated examples of known-good agent behaviors) catch regressions on critical paths. Supplement with production traces converted to permanent test cases — this is the highest-fidelity data you can get because it reflects real usage patterns. LangChain's `agentevals` package (MIT, 650+ stars, Feb 2025) provides ready-made trajectory evaluators for exactly this: you feed it a dataset of example runs with expected tool-call sequences, and it scores how closely the agent follows the reference trajectory. — [langchain-ai/agentevals GitHub](https://github.com/langchain-ai/agentevals)
- **Layer 3 — Layer your evaluators.** No single evaluator type catches everything. Stack three approaches: (1) **Trajectory Match** — hard-coded reference comparison for well-defined workflows, catches regressions in tool selection order and arguments; (2) **LLM-as-Judge** — a second LLM reviews agent output against a rubric, effective for nuanced quality signals that have no ground truth; (3) **Human Review** — subject-matter experts score a sample of production traces, used to calibrate and validate the LLM-as-Judge and surface "metric green, user red" failures. Run all three. — [LangSmith Trajectory Eval Docs](https://docs.langchain.com/langsmith/trajectory-evals), [Confident AI](https://www.confident-ai.com/blog/definitive-ai-agent-evaluation-guide)
- **Layer 4 — Automate the feedback loop.** Wire evals into CI/CD so every prompt change, tool addition, or model swap triggers a test run against the golden dataset. Gate deploys on a minimum reliability score. LangChain's blog frames this as three trigger mechanisms: commit-based (fires on code changes), schedule-based (detects model drift over time), and event-driven (responds to deployment events). AWS Labs' `agent-evaluation` framework (Apache-2.0, 370+ stars) implements this end-to-end: it generates synthetic test conversations via "dueling LLMs" and runs them against agent targets, surfacing failure patterns automatically. — [awslabs/agent-evaluation GitHub](https://github.com/awslabs/agent-evaluation), [LangChain Production Blog](https://www.langchain.com/blog/production-monitoring)

## Evidence

- **LangChain AgentEvals (GitHub, 650+ stars):** Provides `create_trajectory_match_evaluator` for Python and JS/TS. Evaluators run agent trajectories against reference datasets using live LLM models, testing tool selection, response formatting, and prompt modification effects. Companion package `openevals` provides the judge primitives. — [github.com/langchain-ai/agentevals](https://github.com/langchain-ai/agentevals)
- **AWS Labs Agent Evaluation (Apache-2.0, 370+ stars):** Uses "dueling LLMs" to synthesize multi-turn conversational test data, combines with anonymized production traces, and supports Bedrock/Q agents. Integrates directly into GitHub Actions CI workflows. Includes evaluators for success rate, tool call accuracy, and cost efficiency. — [github.com/awslabs/agent-evaluation](https://github.com/awslabs/agent-evaluation)
- **Zalor Agent Test (GitHub Action):** CI integration that exchanges GitHub OIDC tokens (zero secrets required), runs the agent against a dataset on the Zalor platform on every PR, compares results to the current baseline, posts a score delta comment on the PR, and promotes the run to baseline on merge. — [github.com/actions-marketplace-validations/ZalorAI_zalor-agent-test](https://github.com/actions-marketplace-validations/ZalorAI_zalor-agent-test)
- **Google Cloud evaluation taxonomy (Nov 2025):** Organizes agent evaluation along two axes: (1) what to evaluate — agent success/quality, reliability, tool use, safety; (2) how to evaluate — synthetic data, production traces, human-in-the-loop. Explicitly calls out the three-pillar model and recommends blending all three data generation strategies. — [cloud.google.com/blog](https://cloud.google.com/blog/topics/developers-practitioners/a-methodical-approach-to-agent-evaluation)

## Gotchas

- **Metric green, user red.** LLM-as-Judge scores can correlate with human quality at scale but diverge on edge cases that matter to users. Calibrate judges against human-reviewed traces, and always sample production traces for manual spot-checks. A passing score from a judge is not a guarantee. — [Confident AI Guide](https://www.confident-ai.com/blog/definitive-ai-agent-evaluation-guide)
- **Golden datasets go stale.** The behaviors your agent exhibits on day one of deployment will differ from day 90 as users discover new phrasings and edge cases. Re-annotate production traces into test cases continuously — treating evaluation as a one-time setup is a common failure mode. — [LangChain Production Blog](https://www.langchain.com/blog/production-monitoring)
- **Trajectory match is brittle for flexible agents.** If your agent has legitimate variation in how it achieves a goal, strict trajectory matching will flag correct-but-different paths as failures. Use trajectory match for well-bounded workflows (known expected tool sequence), and switch to LLM-as-Judge for open-ended task completion. Mixing the two in the same eval suite leads to confused signal. — [LangSmith Trajectory Eval Docs](https://docs.langchain.com/langsmith/trajectory-evals)
- **Stochasticity means single-run pass/fail is misleading.** An agent that passes a test once may fail the same test on the next run due to LLM non-determinism. Re-run critical scenarios at least 3 times and track aggregate pass rates. Treating single-run results as definitive leads to false confidence before shipping. — [Confident AI Guide](https://www.confident-ai.com/blog/definitive-ai-agent-evaluation-guide)
