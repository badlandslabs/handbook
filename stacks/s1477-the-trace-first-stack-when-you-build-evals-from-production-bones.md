# S-1477 · The Trace-First Stack — When You Build Evals from Production Bones

Your agent shipped with a 94% pass rate on your test set. In production, it silently fails on 23% of tasks — but only some of those failures are captured, and none of them become test cases. Every production failure is a one-time cost unless you wire it back into your eval suite. The teams with reliable agents don't have better models. They have better pipelines for turning production failures into reproducible test cases — and they run those test cases before every agent change.

## Forces

- **Past failures are your best test cases.** A test case derived from a real production failure has perfect recall for that failure mode. A test case written by an engineer predicting failure modes has unknown coverage. Production traces are the highest-signal input to any eval pipeline.
- **Eval tests drift from production distribution.** If your eval suite never updates, it optimizes for the agent passing tests — not for the agent succeeding in production. The gap between eval distribution and production distribution compounds silently until a regression surfaces at the worst time.
- **Not all failures are equal.** A failure that occurs 0.1% of the time and costs nothing is worth less than a failure at 4% that causes data corruption. Weighting eval cases by production frequency focuses regression testing on what actually matters.
- **Synthetic test generation is necessary but not sufficient.** LLM-paraphrased queries and perturbation of existing cases expand coverage, but they can't discover novel failure modes. Only production traces do that — but they arrive one at a time and are easy to lose.

## The move

- **Capture structured failure traces in production.** Log every agent task with its input, all tool calls, tool responses, and final output. Tag each trace with a success/failure signal. At minimum, log the full action transcript — not just the outcome. Anthropic calls this the **Task/Trial/Transcript** model: a task defines inputs and success criteria, a trial is an attempt, and the transcript is the full action trace. Transcripts are what you mine for regression cases.
- **Derive test cases automatically from failure traces.** Extract the input and expected outcome from each confirmed production failure, and add it to your eval suite as a regression test. This requires a human review step — not every logged failure is an eval-worthy case — but the pipeline makes it a routine review task rather than an ad hoc investigation. Tools like Phoenix (Arize), Braintrust, and LangSmith support trace export to eval datasets.
- **Generate synthetic cases from real inputs via LLM paraphrasing.** Take confirmed production inputs and generate paraphrases using an LLM. This expands coverage without the cost of waiting for failures. The KDD 2025 evaluation tutorial calls this **synthetic data generation from historical traces** — a standard enterprise practice. Apply domain-specific perturbation (time shifts for date-sensitive tasks, currency adjustments for financial tasks) to catch brittleness.
- **Weight eval cases by production frequency, not uniformity.** A test case that catches a failure occurring 12% of the time should get more weight in your aggregate score than one that has never failed in production. Track mean trials-to-success as a complement to pass rate — it measures how many retries an agent needs, which matters for latency and cost even when the task eventually succeeds.
- **Run full eval suite in CI before agent changes ship.** Every agent modification — prompt change, tool change, model swap — runs against the full regression suite. This is where the pipeline closes: a production failure becomes a test case, the test case runs in CI, and a future regression fails CI before it reaches production. Anthropic explicitly recommends evals as CI gates.
- **Calibrate graders against human judgment regularly.** Track agreement rates between automated graders and human reviewers over time. Disagreement increasing on specific input types signals that eval criteria have drifted from actual quality expectations — a sign to revisit the grading rubric.

## Evidence

- **Anthropic Engineering:** Defines the Task/Trial/Transcript model for agent evaluation and recommends building evals before the agent is fully built — not after. Describes evals as a communication channel between product managers (defining "good") and engineers (encoding it). Recommends CI-gated evals that run before every agent change. — [Anthropic Engineering](https://www.anthropic.com/engineering/demystifying-evals-for-ai-agents), January 9, 2026
- **KDD 2025 Tutorial (SAP):** Documents evaluation data generation methods including historical production traces as golden datasets, synthetic data generation via LLM paraphrasing, expert annotation, and perturbation of existing cases for robustness testing. Proposes a 2D taxonomy: Interaction Mode (Static/Offline vs. Dynamic/Online) × Metrics Computation (Code-based, LLM-as-Judge, Human-as-Judge). — [SAP / KDD 2025](https://sap-samples.github.io/llm-agents-eval-tutorial/2025_KDD_Evaluation_and_Benchmarking_of_LLM_Agents.pdf)
- **HN post (colinfly):** Documents that evaluating a production agent surfaced primarily software bugs, not model quality issues — broken URLs in tool calls (score dropped to 22), localhost calls in cloud environment, missing API keys causing silent failure, real CVEs flagged as hallucinations (eval issue, not model issue). Concludes that eval catches the entire system: tools, environment, data access, and agent interaction — not just output quality. — [Hacker News](https://news.ycombinator.com/item?id=47416033), March 2026

## Gotchas

- **Capturing traces without tagging outcomes is useless.** You need a binary or graded success signal on every trace, not just the full transcript. Without outcome tagging, you have a corpus of agent behavior with no way to extract failures.
- **Eval tests overfit to the current agent's behavior.** If you only add test cases from production failures of the current agent, you optimize for that agent's quirks. Mix in synthetic cases and cases from different agent architectures to keep the suite general-purpose.
- **Pass rate is a misleading primary metric.** A 95% pass rate means nothing if the 5% failures represent 40% of your production volume. Always pair pass rate with weighted failure frequency by task type.
