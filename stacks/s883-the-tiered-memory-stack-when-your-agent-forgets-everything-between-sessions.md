# S-883 · The Tiered Memory Stack — When Your Agent Forgets Everything Between Sessions

Your agent aced the demo. Twelve sessions later, it greets the same user like a stranger. The conversation history is gone, user preferences are gone, every lesson learned is gone. LLMs are stateless by default — the fix is not one memory store but a tiered architecture that knows what to keep hot, what to page out, and who decides. The Tiered Memory pattern gives agents an OS-style memory hierarchy where the LLM itself is the memory manager, not an external retrieval pipeline.

## Forces

- **Context stuffing hits a wall.** Expanding context windows to millions of tokens sounds like the solution — but BEAM benchmarks show large contexts still fail at contradiction resolution (updating a fact that conflicts with an earlier one) and cost compounds at frontier API prices ($0.01/M tokens means a 600K-token context hits hard on every agent turn)
- **Flat vector stores treat all memory the same.** Episodic facts, durable preferences, procedural knowledge, and session state get one retrieval bucket — but durable facts and short-lived context need fundamentally different treatment; collapsing them into one retrieval layer causes the agent to surface stale preferences and miss active context
- **Retrieval is not the same as remembering.** Standard RAG fetches documents before the model runs — the model has no say in what gets retrieved or when. Self-directed retrieval (Letta/MemGPT model) turns the LLM into both processor and memory manager: it actively decides to search, page results, save information, and update working notes. This is an architectural shift, not a feature add.
- **65% of developers** say AI misses relevant context during refactoring, testing, or code review (Qodo, 2025); senior developers report 19% longer task completion on unfamiliar vs. familiar repositories (METR, 2025) — both findings point to the same root cause: no persistent memory layer

## The move

**Build a three-tier memory hierarchy inspired by MemGPT/Letta's OS-memory model, where the LLM controls what lives where.**

- **Tier 1 — Core Memory (in-context, always hot):** 2–8 KB of essential facts: user identity, active task, current goals. This is the agent's "RAM." Persisted across sessions as a structured object (JSON/dict), injected into system prompt on every turn. The agent reads and writes this directly via tool calls. Budget is tight by design — forcing the agent to decide what actually matters.
- **Tier 2 — Archival Memory (vector store, queried on demand):** Everything that doesn't fit in core but shouldn't be lost. Searched via embedding similarity on explicit `archival_memory_search` tool calls made by the LLM. No automatic stuffing — the agent decides what goes in and when to query. Backed by pgvector, Chroma, Qdrant, or Weaviate. Size limit: none, bounded by storage.
- **Tier 3 — Recall Memory (conversation log, pageable):** Full conversation history, retrieved in chunks when the agent explicitly needs past context. Not in context by default — fetched on demand. The agent pages this in the same way an OS pages RAM: a few recent turns visible, older turns paged out until needed.
- **LLM as the memory manager:** The agent calls `core_memory_replace`, `archival_memory_search`, `archival_memory_insert`, and `recall_memory_search` as actual tool actions — not preprocessing steps done by an external pipeline. This means the model decides what's worth remembering, what conflicts with prior facts, and what to forget. The quality of memory management becomes a property of the model's reasoning, not a property of the retrieval pipeline.
- **Structured facts over embeddings where precision matters:** For durable facts (preferences, constraints, identities), store as structured key-value pairs, not just embedding vectors. This enables exact-match lookups and conflict detection — the Memory Guardian approach (open-source) detects conflicting facts across retrievals and flags them for resolution.
- **Memory governance layer on top:** Stale memory, fact decay, and retrieval explainability are real problems at scale. Add TTLs on episodic memories, conflict detection on durable facts, and log retrieval decisions so you can audit why the agent retrieved something.

## Gotchas

- **Auto-summary is a lossy operation.** Summarizing long context to fit a smaller context window discards detail — the agent may lose specific constraints or preferences it encoded in earlier turns. Treat summarization as a last resort, not the primary memory mechanism.
- **Embedding-based retrieval misses exact matches.** If you store "user prefers dark mode" as an embedding, a query for "light mode preferences" may surface the dark mode fact incorrectly. For durable facts, combine vector retrieval with structured key-value lookups.
- **The agent may not know what it doesn't know.** A model that doesn't realize it should check its memory won't call the retrieval tools. Prompt the agent explicitly to check archival and recall memory at session start and on task handoffs — make memory retrieval a first-class action, not an optional one.
- **Cross-agent memory is unsolved for most teams.** When multiple agents work on the same user, each may have its own memory store. User preferences can diverge across agent instances. Per-user, per-agent memory isolation helps initially but creates consistency problems. Design for merge strategies early.
