# S-1521 · The Tool Sovereignty Stack — When Your Agent Has Too Many Tools

You've built a capable agent. You connected it to your code repository, your database, your Slack, your GitHub, your Stripe account, your internal wiki. Now your agent has 200 tools, the MCP protocol has broadcast every one of them to every client, and your first prompt costs 108,000 tokens before the user has said anything. The tools work. The agent is drowning.

## Forces

- **Tool definitions are the new context tax.** A single Docker MCP server can consume 125,000 tokens in tool definitions. Add GitHub, Jira, Figma, Notion, and a dozen custom tools, and you've burned 40–60% of your context window before the conversation starts. This isn't a configuration mistake — it's what happens when tool count scales past 20–30.
- **More tools means more attack surface.** MCP's `tools/list` is mandatory. Every client that connects sees every tool your agent ever had, every parameter schema, every resource access path. The Snowflake Cortex sandbox escape (March 2026), the Alibaba cryptomining agent pivot, and CVE-2025-59528 (CVSS 10.0, Claude Code worktree attack enabling home-directory write and root-level code execution) all trace back to the same root cause: an agent with more tool access than it needed, and an environment that didn't scope that access tightly enough.
- **Tool selection accuracy degrades at scale.** Internal testing at Anthropic showed Opus 4 solving only 49% of MCP-tool-usage tasks when all tools were pre-loaded. After implementing on-demand tool discovery, accuracy jumped to 74%. With 200 tools, the model spends more tokens reasoning about which tool to call than actually doing useful work.
- **Sandbox isolation is not optional for code execution.** OWASP Agentic Top 10 item ASI05 ("Unexpected Code Execution") mandates sandboxing as a control, not a recommendation. Standard containers share the host kernel and are insufficient for agentic workloads executing LLM-generated code. The three dominant 2026 isolation technologies are Firecracker microVMs (strongest, for regulated data), gVisor (syscall-level, for compute-heavy multi-tenant), and V8 Isolates (JS-only, for lightweight latency-critical tasks).

## The Move

Sovereignty over your agent's tools means three things in concert: **scope tightly, load lazily, and isolate hard.**

- **Curate tool scope before connecting more.** Not every capability should be a tool. Group related operations into a single tool with parameters rather than exposing 15 individual API calls. Composio (29k+ GitHub stars) surfaces 100–200 integrations, but teams that connect all of them report context bloat — the value is in having the catalog available, not all of it loaded at once.
- **Load tools on-demand, not at startup.** Anthropic's MCP Tool Search feature (shipped in Claude Code 2.1.7, January 2026) reduced initial context consumption by 85% by deferring tool definition loading until the model actually needs them. A Docker MCP server that previously consumed 125k tokens now loads zero tokens until the agent actually needs Docker commands. This is the production-ready pattern: `enable_tool_search: true` in Claude Code configuration.
- **Scope sandbox credentials to the task, not the environment.** The Google Antigravity sandbox escape (CSA report, April 2026) and CVE-2025-59528 both succeeded because the agent had access to secrets and filesystem paths that its task didn't actually require. AWS Bedrock AgentCore Code Interpreter (August 2025) and OpenAI's April 2026 Agents SDK sandbox update (supporting E2B, Modal, Docker, Vercel, Cloudflare, Daytona, Runloop, and Blaxel) both isolate credentials from the execution environment — the agent runs code in a container that cannot see the API keys used to create it.
- **Pick your browser automation tool by workflow predictability.** For known-selector flows and structured scraping: Playwright MCP (Microsoft, 34k+ GitHub stars, 60M+ weekly downloads). For multi-step navigation without fixed paths where the agent must reason about page state: Browser Use or Stagehand (DOM-element-aware, auto-caching reduces repeated workflows toward Playwright-native performance). For quick URL read/extract/scrape with no local Chromium: CRW scrape API or Browserbase cloud browser.
- **Use Docker MCP Gateway as the orchestration layer for multi-container agentic apps.** Docker's MCP Gateway (2026) lets you define and manage MCP servers declaratively, connecting agents to external tools via `docker compose` configuration. This is the emerging pattern for teams that want tool access defined in infrastructure-as-code rather than scattered across SDK initialization.

## Gotchas

- **Connecting every Composio integration is the same mistake as connecting every MCP server.** The value of large tool catalogs is availability, not activation. Load only what the agent's current session needs; maintain the catalog for when tasks shift.
- **Docker containers alone do not constitute a sandbox.** Container isolation shares the host kernel. For any agent that executes LLM-generated code (code interpreter, autonomous scripting, tool-building), use Firecracker microVMs or gVisor at minimum. Standard `docker run` is not sufficient.
- **MCP's `tools/list` cannot be suppressed.** The protocol requires it. If your agent connects to an untrusted MCP server, the server learns your entire tool inventory. Treat MCP server access with the same network-level isolation you would a third-party API — VPC, egress filtering, and principle of least privilege.
- **Tool accuracy improvements from lazy loading plateau.** At 85% token reduction, accuracy gains are significant. At 95%+, you start losing the cross-tool reasoning that comes from the model having ambient awareness of available capabilities. Test your specific tool count and benchmark accuracy curves to find the right trade-off.
