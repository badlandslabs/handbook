# S-1284 · The Quadratic Burn — When Token Costs Compound Faster Than Results

Your agent completed the task. The output is correct. The user is happy. And your monthly bill is 100x what you projected. The problem isn't that your model is expensive — it's that your agent's cost compounds with every step. Each tool call adds tokens. Each turn resends the accumulated context. What started as a 5,000-token task is 80,000 tokens by step 20, and nobody noticed until the invoice arrived. This is the quadratic burn pattern, and it is the most common cost failure in production agentic systems.

## Forces

- **Agents resend full context every turn.** Unlike a stateless API call, every step in a plan→tool→observe→reflect loop re-transmits the entire conversation history. Costs grow super-linearly with step count — approximately N² in naive implementations. (Exe.dev Blog, March 2026 — https://blog.exe.dev/expensively-quadratic)
- **Context bloat compounds silently.** Measured production data shows 24 memory entries add ~594 tokens per call; 500 entries add ~8,000 tokens. The agent doesn't surface this — your billing dashboard does, two weeks later. (Mem0 via NiteAgent, May 2026 — https://niteagent.com/blog/ai-agent-cost-optimization-2026)
- **Demand elasticity inverts savings.** Cheaper per-token pricing increases total spend, not decreases it. Lower costs unlock more ambitious tasks, more agents, more steps — driving total inference spend upward even as unit prices fall. (AnhTu.dev, May 2026 — https://anhtu.dev/token-economics-cost-optimizing-ai-agents-production-2026-2257)
- **The one-model-for-everything trap.** Frontier models (e.g., $10/M input) vs. flash models (e.g., $0.10/M input) represent a 100× price difference for equivalent simple tasks. Most teams default to the same model regardless of task complexity, overpaying by 40–85%. (LockLLM, 2026 via NiteAgent)
- **Runaway loops are invisible until they're expensive.** A stuck agent doesn't crash — it keeps calling the LLM, each time with a growing context window. One documented case: a LangChain agent looped undetected for 11 days, burning $47,000 on a pipeline budgeted at under $200/month. (AgentMarketCap, April 2026 — https://agentmarketcap.ai/blog/2026/04/12/ai-agent-token-consumption-gap-enterprise-agentic-workloads)

## The Move

The cost engineering pattern has four levers, applied in order of impact:

**1. Prompt caching — the highest-leverage fix.** The system prompt and all tool schemas are the fixed prefix sent on every turn. Anthropic's cached prompt reads cost 90% less than regular input tokens. Make this prefix the primary caching target: extract every static element (instructions, tool definitions, few-shot examples) into a cached prefix layer. A single agent with 10 tool definitions and a 1,500-token system prompt saves thousands of dollars per month at even modest query volume. (Anthropic API pricing via Zylos Research, May 2026 — https://zylos.ai/en/research/2026-05-02-ai-agent-cost-engineering-token-economics)

**2. Hard budget circuit breakers — non-negotiable.** Set a per-task token limit and a per-session dollar cap enforced at the proxy layer, not in the agent logic. The agent cannot self-limit: a looping agent has no awareness it's burning budget. Circuit breakers must sit in the infrastructure layer (an LLM proxy like LiteLLM, or a guardrail tool like reivo-guard) so they survive agent prompt injection or logic errors. A runaway agent that costs $15 in 10 minutes will cost $340 in one hour without a hard stop. (aicost.ai Agent Loop Calculator, 2026 — https://aicost.ai/ai-cost-guides/calc/agent-loop-cost)

**3. Model routing by task complexity.** Classify task types into tiers — extraction/routing (cheap, fast model), reasoning/synthesis (mid-tier), creative or edge-case handling (frontier). Route dynamically: simple classification tasks that route to the right agent don't need Opus-tier reasoning. A real-world routing implementation cuts average per-task cost by 40–85% with no measurable quality degradation on tier-appropriate tasks. (LockLLM via NiteAgent; vLLM Semantic Router, NeurIPS 2025 — https://github.com/vllm-project/semantic-router)

**4. Token budgets per step — force the agent to think cheaply.** Before each tool call, check remaining budget. Define three budget actions: continue normally, force synthesis (wrap up with what's known), or hard stop (return with emergency summary). The agent that would otherwise spend 50 steps doing thorough research gets cut off at a cost-appropriate threshold and delivers a "good enough" result instead of a "perfect but expensive" one. (Neel Mishra, agent cost management pattern, 2026 — https://neelmishra.github.io/blog/mlops/llm-agents/agent-cost-management.html)

## Gotchas

- **Prompt caching has a minimum prefix length and TTL.** Anthropic's prompt caching requires at least 1,024 tokens in the cached prefix. Shorter system prompts don't benefit. TTL limits (typically 5–10 minutes) mean high-latency workflows lose cache hits.
- **Budget circuit breakers can trigger mid-task.** If a complex task hits the budget cap at step 18 of 20, you get an incomplete deliverable. Build graceful degradation: force synthesis at 80% budget, hard stop at 100%, so the agent wraps up rather than crashing.
- **Model routing introduces latency.** Classifying task complexity before routing adds 1–50ms (keyword routing is fast; embedding-based routing is slower). For latency-sensitive agents, cache the routing decision alongside the task classification.
- **The $87K Singapore fintech case is a lower bound.** That was one LangChain pipeline. Uber burned through its full 2026 AI budget in four months of employee Claude Code usage, per HN reporting (June 2026). Individual teams may not see aggregate spend across users, agents, and sessions. (Hacker News — https://news.ycombinator.com/item?id=48375544)
- **Caching tool results is harder than caching prompts.** Tool outputs are dynamic, not static. Semantic caching (embedding-based lookup for "similar enough" tool results) works but adds complexity. Deterministic caching (exact match on tool name + args) is simpler but hits less often. Butter's tree-structured approach is the most production-proven pattern for this.
