# S-1315 · The Invisible Grader Stack — When You Cannot Prove Your Agent Is Working

You shipped the agent. The demo was perfect. Now it runs in production, handles real users, and you have no idea if it is getting better or worse. Traditional software testing — pass/fail, coverage %, error rate — does not apply. Your agent sometimes succeeds through the wrong reasoning, sometimes fails through the right reasoning, and the only signal you have is a satisfied customer or a confused one. This is the invisible grading problem: you built the system but never built the test.

## Forces

- **The correct answer is not evidence of correct reasoning.** An agent tasked with inventory reporting returns the right number — but by reading last year's file instead of querying the live system. The output is right. The behavior is wrong. Classical metrics reward outcomes, not competence.
- **Benchmarks are gaming targets, not truth.** UC Berkeley researchers demonstrated in April 2026 that a single automated scanning agent achieved 100% on eight prominent benchmarks — SWE-bench, WebArena, Terminal-Bench, FieldWorkArena, and others — without solving a single task, by exploiting evaluation infrastructure flaws (parser overwrites, config leakage, DOM injection). Leaderboard scores are measuring "ability to read answer keys," not "ability to solve problems."
- **The evaluation lag problem.** By the time you know your agent regressed, users have already experienced the regression. Static test suites validated yesterday's behavior against yesterday's inputs — they do not catch the new failure mode that emerged overnight.
- **LLM-as-judge is seductive and unreliable.** Using one LLM to evaluate another is easy to set up and hard to validate. A judge that marks a refund-confirmation response as "helpful" while the agent never actually processed the refund is a passing score that means nothing.
- **Trajectory vs. outcome tension.** Evaluating only final output misses the process. Evaluating every step generates massive data and still requires human judgment to determine if a step was warranted.

## The Move

Shift from static benchmark scores to a continuous, multi-dimensional evaluation pipeline that treats agent quality as a production discipline, not a launch-day checkbox.

- **Evaluate trajectories, not just answers.** Instrument every tool call, reasoning step, and decision point. An agent that reaches the correct answer via the wrong path has demonstrated a failure mode worth catching — even if it "passed." Trace-level analysis catches silent failures (correct output, incorrect process) that outcome-only metrics miss.
- **Layer code evaluators and LLM judges by function.** Use deterministic checks — schema validity, exact match, regex, latency, token count, tool name — for what can be programmatically verified. Reserve LLM judges for semantic checks: correctness, helpfulness, safety, tone, task completion, tool-call appropriateness. Do not use a judge model for what a regex can check.
- **Prefer boolean or categorical labels over numeric scores.** A 1–10 scale invites gaming and has no meaningful floor. Boolean (correct/incorrect) or categorical (correct, partially-correct, incorrect, insufficient-evidence) are actionable. Add `insufficient_evidence` as a category rather than forcing a binary decision on cases the judge cannot assess.
- **Build golden datasets from three sources:** anonymized production logs, LLM-generated synthetic conversations (dueling LLMs where one role-plays the user), and human-curated interactive sessions saved from trace logs. Each source covers different failure modes — production data captures real patterns, synthetic data fills coverage gaps, human sessions encode edge cases.
- **Validate judges against human labels before gating on them.** Run the judge on a sample annotated by humans, measure agreement. A judge that disagrees with SMEs 40% of the time should not gate deployments. Track judge behavior over time — judges drift just like models do.
- **Close the eval loop with production monitoring.** Route traces with low confidence scores or anomaly signals to SMEs for annotation. Encode institutional knowledge from those annotations back into evaluators. Every expert departure that is not annotated takes domain knowledge with it.
- **Track operational metrics as first-class evaluation targets.** Latency, cost per task, token efficiency, tool reliability, and policy compliance determine whether a technically capable agent is viable at enterprise scale. An agent that scores 95% on task accuracy but runs 3× slower than a human and costs $0.40 per query is not passing its evaluation.

## Gotchas

- **Ditch the 1–10 scale.** Numeric scores are easy to game and hard to interpret. Boolean or categorical labels with an `insufficient_evidence` option are more actionable and less prone to false confidence.
- **Do not gate on an unvalidated judge.** If you have not measured the judge's agreement rate against human labels, you do not know if it is better than a coin flip. Run the validation before the judge makes decisions that matter.
- **Coverage is not the same as correctness.** An agent that succeeds on 80% of test cases and fails silently on the remaining 20% is not an 80% agent — it is a broken agent with a misleading dashboard score. Focus on the failure mode distribution, not the aggregate.
- **Golden datasets rot.** Inputs that worked six months ago may no longer match production patterns. Re-annotate and refresh test suites from live traces quarterly at minimum, or the eval suite will drift from reality.
- **Cost and latency are evaluation signals, not just operational concerns.** An agent that passes every quality metric but costs $2 per query or takes 45 seconds per task is not production-viable. Include operational constraints in the eval definition from day one.
