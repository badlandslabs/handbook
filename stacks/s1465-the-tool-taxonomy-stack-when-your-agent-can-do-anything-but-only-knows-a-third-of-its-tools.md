# S-1465 · The Tool Taxonomy Stack

[When you ship an agent with 40 tools and it calls the same 3 over and over — or worse, you gave it 3 tools and it needed 30. The question is never "how many tools" but "which ones, in what configuration, with what interface."]

## Forces

- **Tool supply has outpaced tool demand.** The MCP registry grew 36× in 13 months — from ~4,888 tools in January 2025 to 177,436 by February 2026 (Stein, UK AI Security Institute / Oxford, arXiv:2603.23802v1, March 2026). Agents now live in a tool-rich environment they've never been designed to navigate efficiently.
- **The action-to-observation ratio has inverted.** Action tools (tools that modify external state: execute code, write files, call APIs, send messages) now represent 65% of MCP tool downloads, up from 27% in early 2025. Agents are no longer passive observers — they're becoming active operators. This changes the tool design question fundamentally.
- **The "what tools do I give it" question is underspecified.** Teams conflate tool count with capability and tool specificity with coverage. Both mistakes are expensive.
- **Tool design shapes agent behavior more than model choice.** A poorly described `search_web` tool produces different results than a well-scoped `fetch_tech_news_articles(topic, days=7, limit=20)` — even on the same model. This is underappreciated.
- **28% of MCP servers are AI-coauthored.** 36% of MCP tools were partially generated by AI (Stein, 2026). The tools themselves are now products of the agents they're meant to serve — a feedback loop that is reshaping the ecosystem faster than standards can track.

## The move

Map the agent's tool surface not by enumerating capabilities but by targeting the actual task topology. A 5-step research task needs different tools than a 50-step DevOps pipeline.

### 1. Categorize by interaction type, not function name

The 177K-tool dataset reveals a consistent taxonomy: **observation tools** (read, search, fetch, monitor), **action tools** (write, execute, send, call, create, delete), and **reasoning tools** (evaluate, compare, rank, score). Most agents are starved for reasoning tools and over-equipped with action tools. Add evaluation and judgment capabilities before adding more action surface.

### 2. Use MCP as the interface contract, not the capability ceiling

MCP (Model Context Protocol) is now the dominant standard — Microsoft, Anthropic, GitHub, and most major agent frameworks have adopted it. The practical value: you configure a tool once and any MCP-compatible agent (Claude Code, Cursor, Continue, custom builds) can use it. But MCP describes the *interface*, not the *quality* of the underlying tool. A well-described `file_read(path, lines)` beats a vague `read_file`. Audit tool descriptions the way you'd audit an API contract.

### 3. Start with the minimum viable toolset, expand by failure mode

Teams that start with large tool inventories report agents that loop, call wrong tools, or ignore available tools entirely. The empirical pattern from production deployments: begin with 3–5 tools covering the core loop, then add tools only when the agent fails a recognizable task class. Use a tool request log to drive expansion, not upfront design.

### 4. For browser-based agents: use the right abstraction layer

Browser automation is the dominant interface for production agents (web scraping, research, form submission, UI testing). Three layers exist:

- **Playwright** (raw): Full cross-browser control, highest flexibility, most boilerplate. Best for deterministic multi-step workflows.
- **Stagehand** (by Browserbase): Playwright underneath, agent-friendly `act`/`observe`/`extract` methods. Reduces LLM-to-browser glue code by 60–70% vs raw Playwright. Best for teams building agent-native browser interactions.
- **MCP browser servers** (standardized): Configure once, use across agents. Vercel `agent-browser` produces compact page representations (DOM snapshots vs screenshots) to reduce token cost. Best for multi-agent deployments where multiple agents share browser access.

### 5. Reverse-engineer tools from the target environment

The most effective production tool sets aren't designed from scratch — they're derived by watching how users or applications already interact with the target system. The Show HN "reverse-engineering web apps into agent tools" approach (HN, July 2026) trains agents by observing authenticated web app API calls: the agent runs inside the app, watches its own backend calls, and generates tools that replicate the app's own patterns. This produces tools that are maximally contextually valid — the agent uses the app the way it's designed to be used.

### 6. Monitor tool call distribution, not just task success

Tool call logs reveal the real toolset. If 3 tools represent 80% of calls on a system with 30 tools, the other 27 are dead weight — adding context cost without contributing to outcomes. Prune tools with <2% call share after 100+ task completions. This is cheaper than it sounds and more valuable than most teams expect.

## Gotchas

- **Tool count ≠ capability.** Adding more tools increases context overhead (tool definitions consume context window) without proportional capability gains. The S-1462 "tool tax" pattern applies here — every tool has a cost that must be justified.
- **MCP doesn't guarantee tool quality.** The protocol standardizes the interface, not the tool's description, error handling, or output format. Audit each tool's schema and response quality the same way you'd audit a third-party API.
- **Browser screenshots are token-expensive.** Full-page screenshots on complex web apps can consume 10–30K tokens per frame. DOM snapshots, wireframe representations, or targeted element extraction are dramatically cheaper. If token cost is a concern, screenshot-only browser tools will become a bottleneck.
- **AI-coauthored tools can encode AI-blind spots.** Tools generated by LLMs tend to reflect patterns the LLM knows well and miss edge cases, unusual error states, or domain-specific invariants. Validate AI-generated tool schemas against real production inputs before shipping.
- **Cross-agent tool sharing requires version discipline.** When the same MCP server is used by multiple agents, a breaking change to a tool definition silently breaks all of them. Pin tool versions and treat MCP server updates as you would a library dependency — with a test run before rollout.
