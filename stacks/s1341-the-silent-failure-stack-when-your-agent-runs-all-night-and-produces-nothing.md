# S-1341 · The Silent Failure Stack — When Your Agent Runs All Night and Produces Nothing

Your agent is running. No errors. No crashes. Logs look healthy. Monday morning you find it made zero useful progress for nine hours, burned through $2,300 in API calls, and left three downstream systems in inconsistent states. The monitoring dashboard showed green the entire time. The agent never raised an exception — it just kept doing the wrong thing confidently. This is the silent failure problem, and it is the defining operational challenge of production agentic systems.

## Forces

- **Agents fail without raising exceptions.** Conventional software crashes and logs a stack trace. An agent can behave incorrectly — looping, hallucinating tool parameters, approving bad outputs — while every system metric reads nominal. The absence of an error is not evidence of success.
- **The safety net creates the trap.** `max_iterations` and `try/catch` wrappers are the standard advice for preventing infinite loops. They work — until they create a worse failure mode: agents entering dead-end states that look like progress, never triggering the guard, never escalating, just accumulating cost silently.
- **The observability gap.** LangSmith, LangFuse, Arize, and Helicone all show traces: latency, token counts, span trees. They answer what happened. They do not answer whether the agent is producing business value right now, whether it is degrading silently, or whether you will learn it broke before your users do. (DEV Community / ceaksan, citing HN reports of monitoring tools showing green for two-plus years while agents looped — https://dev.to/ceaksan/an-ai-agent-got-stuck-in-a-loop-the-monitoring-tools-saw-nothing-1ai)
- **Failure propagates across layers.** A single tool parameter hallucination can cascade through planning, memory, and action modules. Error propagation — not individual errors — is the central bottleneck in robust multi-step agents. (Zylos Research, January 2026 — https://zylos.ai/research/2026-01-12-ai-agent-error-handling-recovery/)

## The Move

A layered defense stack that addresses each silent failure mode at the layer where it is cheapest to catch:

- **Self-describing tool responses.** Every tool returns a structured result with explicit `status` field — `complete`, `failed`, `needs_revision` — not ambiguous empty arrays or HTTP 200 with bad data. The agent must always know whether it is done. (AIWave Blog, July 2026 — https://aiwave.hashnode.dev/why-your-ai-agent-keeps-looping-and-how-to-fix-it-a-deep-dive-into-react-pattern-failures)
- **Three-tier loop detection.** Exact-repetition loops caught with O(1) call hashing. Near-identical loops caught with cosine-distance embedding similarity on consecutive tool call sequences. Behavioral loops caught with progress-tracking — a lightweight state vector that records whether the agent's actual outputs are changing, not just whether it is calling tools. (SupraWall / Wiserautomation, 2026 — https://www.supra-wall.com/learn/ai-agent-infinite-loop-detection)
- **Circuit breaker with semantic fallback.** When a tool or model call fails N times, the circuit opens and a lower-capability fallback path takes over — not just a retry loop. The fallback preserves partial state rather than restarting from zero. (Zylos Research, January 2026; Preporato, May 2026 — https://zylos.ai/research/2026-01-12-ai-agent-error-handling-recovery/, https://preporato.com/blog/error-handling-resilience-patterns-agentic-ai-systems)
- **Supervisor tree for multi-agent handoffs.** Each agent in a multi-agent pipeline has a lightweight supervisor that monitors liveness, token budget, and output quality. The supervisor, not the agent itself, decides when to kill and restart. Inspired by Erlang/OTP supervisor trees and applied to AI agents in the orchestration-playbook repo. (p3nchan/orchestration-playbook, MIT, March 2026 — https://github.com/p3nchan/orchestration-playbook)
- **Business-outcome monitoring, not trace monitoring.** A lightweight semantic eval that runs on every Nth production trace: does this output advance the stated goal? Is the output consistent with prior steps? Attach this as a canary metric to the trace dashboard, not as a separate eval job. It closes the "green dashboard, broken agent" gap. (DEV Community / ceaksan, 2025 — https://dev.to/ceaksan/an-ai-agent-got-stuck-in-a-loop-the-monitoring-tools-saw-nothing-1ai)

## Evidence

- **Production incident report (Asynq.ai / Modelia.ai):** Candidate evaluation agent hallucinated tool parameters, got stuck in loops, and produced contradictory evaluations. Image generation agent approved flawed outputs. Combined effect: 3x budget overrun, zero alerts triggered. Fixed with structured tool schemas, conversation history windowing, and explicit loop detection. (Harsh Rastogi, March 2026 — https://harshrastogi.tech/blog/agentic-ai-error-recovery-observability-patterns)
- **$47,000 case study:** A 4-agent LangChain A2A pipeline for market research went from $127/week (Week 1) to $18,400/week (Week 4) without any alerting. Root cause: agents looping on ambiguous tool results, no cost circuit breaker, no progress tracking. Pulled and rebuilt with per-agent spend limits and explicit completion signals. (Kusireddy, Towards AI, October 2025 — https://pub.towardsai.net/we-spent-47-000-running-ai-agents-in-production-heres-what-nobody-tells-you-about-a2a-and-mcp-5f845848de33)
- **Academic taxonomy:** A 2025 arXiv study (arxiv.org/html/2603.06847v1) of agentic AI faults categorizes failures into four types — specification (42% of failures), execution, coordination, and memory — with distinct detection and recovery requirements for each. The ALAS framework pairs each agent with a fault-tolerant compensator for coordinated recovery. (Zylos Research, citing arXiv 2603.06847 and ALAS framework arXiv 2505.12501 — https://zylos.ai/zh/research/2026-05-06-agent-self-healing-failure-recovery)

## Gotchas

- **`max_iterations` is a blunt instrument.** It stops the agent from looping, but leaves it in a dead-end state with no signal that it is stuck. The dead-end is often harder to detect than the loop because nothing looks wrong.
- **Empty results are not failure signals.** An agent calling `search_database(query)` that returns `[]` will often retry rather than conclude "no results found." Make the absence of results an explicit `status: complete` with a `result_count: 0` field, not a null payload.
- **LLM-generated safety patches create local fixes for global problems.** When an LLM is asked to fix a looping agent, it typically patches the immediate symptom — adding a new state like `UPDATE_NEEDED` — without reasoning about whether that state has outbound transitions or whether it creates a new cycle. Dead-end state traps are caused by well-intentioned LLM-written safety patches. (DEV Community, Praveen Tech World — https://dev.to/youngones/preventing-infinite-loops-in-llm-agent-pipelines-the-dead-end-state-trap-pl4)
- **Single-agent systems with proper engineering can match multi-agent accuracy.** Princeton NLP found single agents matched or outperformed multi-agent systems on 64% of benchmarks with the same tools — at half the cost and a fraction of the failure surface. More agents multiply failure modes without proportional accuracy gains. (beam.ai, citing Gartner and Princeton NLP — https://beam.ai/agentic-insights/multi-agent-orchestration-patterns-production)
