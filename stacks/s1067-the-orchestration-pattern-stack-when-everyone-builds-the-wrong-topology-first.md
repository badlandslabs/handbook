# S-1067 · The Orchestration Pattern Stack — When Everyone Builds the Wrong Topology First

The first multi-agent system most teams build is sequential. A → B → C. It works for the demo. Then the product team adds branching logic, a second user type needs a shortcut past B, and the overnight load test reveals that C is the bottleneck for 40% of queries. You didn't pick the wrong framework. You picked the wrong *topology*. The orchestration pattern — how agents are connected, routed, and sequenced — determines reliability, cost, latency, and debuggability as much as model selection or prompt engineering. And the most natural choice is usually the wrong one for production.

## Forces

- **Sequential is the obvious first move.** Linear chains feel correct: "do this, then that." They map to business processes teams already understand. But they have no branching, no conditional routing, and no recovery path when a mid-chain agent fails — the entire pipeline stops.
- **Multi-agent failure is a coordination failure, not an agent failure.** 37% of multi-agent failures trace to inter-agent wiring, not the individual agents themselves. The agents may be fine. The topology is the problem.
- **Pattern choice is a one-way door.** Migrating from sequential to hierarchical, or from fan-out to router, requires rearchitecting the state machine. Teams that pick wrong early pay 4–8 weeks of rewrite.
- **Cost compounds across agents.** A 4-agent orchestrator-worker workflow costs $5–8 per complex task. The wrong topology wastes tokens on agents that should never have been invoked for that query class.
- **Observability precedes debugging.** 89% of teams have observability but only 52% have evals. For multi-agent systems, this gap means failures silently cascade across agent boundaries.

## The Move

Choose your orchestration pattern by answering three questions in order: Does the task have conditional branches? Does the work decompose into independent parallel tasks? Does a single agent hit a ceiling?

**Start with the simplest pattern that fits your answer to those three questions:**

- **Sequential Pipeline (A → B → C):** Fixed linear order. Use only when dependencies are strict, order matters, and no task can begin before the previous one finishes. Example: research → draft → review → publish.
- **Parallel Fan-Out (A → [B, C, D] → E):** One agent broadcasts to multiple specialists simultaneously. Use for tasks where work is truly independent — aggregating search results from different sources, processing multiple files in parallel, running parallel LLM judges. The join step (E) must handle partial failures.
- **Router (A → {B or C or D}):** A classifier or LLM routes each query to the right specialist. Use when task types are categorically different and you want to avoid burning compute on unnecessary pipeline depth for simple queries. Emerging 2026 pattern: difficulty-aware dynamic routing where a lightweight classifier estimates query complexity and allocates pipeline depth proportionally.
- **Hierarchical / Orchestrator-Subagent (Supervisor → [B, C, D]):** A central orchestrator manages conversation state, delegates to specialists, synthesizes results. The orchestrator handles "should another agent be involved?" — specialists focus on execution. Use when task domains are clearly separable (e.g., a sales copilot with a lead-scoring agent + proposal-generation agent + follow-up agent). The orchestrator is a state machine, not a general-purpose agent.
- **Evaluator-Optimizer Loop (A → evaluate → B → evaluate → C → ...):** Agents iterate on output until a quality threshold is met. Use for code generation, creative writing, and optimization tasks where "good enough" is defined by a measurable criterion. Requires a termination condition — without it, loops run until the token budget expires.

**Most production systems combine patterns.** A hierarchical orchestrator might use sequential for each specialist's internal steps, fan-out for parallel tool calls within a step, and a router to decide which specialist gets the task first.

**The MCP factor:** As of 2026, Anthropic's Model Context Protocol has 97M+ monthly SDK downloads. LangGraph + MCP is the dominant enterprise combination for tool orchestration. The protocol matters because agent-to-tool communication is the most frequent inter-system call in any topology.

## Gotchas

- **The "sequential by default" trap.** Teams use sequential because it mirrors their process diagrams. Production systems with conditional logic, variable task complexity, or parallel work streams need router or hierarchical patterns. Sequential is almost always wrong for anything beyond a two-step transformation.
- **Untyped handoffs kill multi-agent workflows.** Every agent-to-agent boundary needs a validated schema with version numbering. Without it, a schema change in agent B silently breaks agent A's output format, and the failure manifests 3 steps later as hallucinated parameters.
- **The join step is where fan-out systems break.** When one of N parallel agents fails, the aggregator must handle partial results gracefully. Production teams report this as the most common silent failure mode in fan-out topologies.
- **Cost modeling comes after topology, not before.** Teams that scope architecture around budget first end up with a topology that fits the budget rather than the problem. Model cost after picking the right pattern — $5–8 per complex task for a 4-agent workflow is acceptable if the task complexity justifies it.
- **Framework coupling is a future rewrite.** LangGraph is the production default in 2026, but the framework landscape shifted significantly in 18 months (AutoGen entering maintenance, CrewAI rewriting their API in v0.98+). Prefer patterns over framework features — the wiring should be replaceable without rearchitecting the topology.
