# S-1328 · The Noisy Judge Stack — When Your LLM-as-Judge Has a 50% Error Rate and You Ship Anyway

You built a "golden dataset" of 200 agent traces. You have an LLM judge grade them against your rubric. The judge says your agent scores 87%. You ship. Three weeks later, users report consistent failures on a case the judge called "excellent." You've been running production on a 50% error rate, and you didn't know it — because the judge never told you it was guessing.

This is the **Noisy Judge Stack** — a production pattern for deploying LLM-as-judge evaluation without being deceived by it. The key insight from research is that noisy is not worthless: an LLM judge with >50% error rate still produces useful *signal* when you aggregate over many cases, use bias mitigation at the input level, and calibrate outputs against a small human-labeled set. The trap is treating it as a ground-truth scorer rather than a probabilistic signal.

## Forces

- **LLM judges have systematic biases that look like confidence.** Position bias (favoring earlier responses), length bias (preferring longer outputs), and agreeableness bias (over-accepting without critical pushback) combine to produce confident-looking scores that correlate poorly with actual quality.
- **A single judge call is not reliable.** Research shows error rates exceeding 50% for LLM judges on non-trivial inputs. A single pass is not a measurement — it's a noisy data point.
- **Calibration is the missing step most teams skip.** Without a small human-annotated subset to anchor against, you have no idea whether your judge's "85%" means 85% or 45% actual agreement with ground truth.
- **The taste gap.** As one HN practitioner put it: "It's easy to get an LLM to give praise, and it's easy to get an LLM to give criticism, but getting an LLM to praise good things and criticize bad things is currently impossible for non-trivial inputs." LLMs lack the aesthetic and domain taste that lets human experts know when something is *actually* wrong versus merely different.
- **Deterministic checks are underrated.** Code-based graders (regex, schema validation, exact-match assertions) cover 60–80% of failure modes at a fraction of the cost and with 100% reproducibility. Teams over-invest in LLM-as-judge and under-invest in the things that can be checked programmatically.

## The Move

Build a three-tier grader stack: deterministic checks for what can be mechanically verified, LLM-as-judge with bias mitigation for nuanced assessment, and human spot-checks for calibration. Use ensemble aggregation over single calls. Tie every score to a trace so a low score traces back to a specific span.

### Grader Selection Heuristic

1. **Can it be checked with code?** Use a code grader. Schema validation, argument type checking, retrieval precision at k, tool call count bounds. Fast, cheap, deterministic. Covers ~60–80% of failure modes.
2. **Is it nuanced but verifiable after the fact?** Use an LLM judge with bias mitigation. E.g., response quality, reasoning coherence, plan soundness.
3. **Is it subjective, safety-critical, or domain-expert dependent?** Use a human grader — but only for calibration (~10–20 cases), not for bulk scoring.

### LLM Judge Bias Mitigation

- **Ensemble across judges:** Deploy 3–5 judge instances with randomized response order. Calculate majority vote. This reduces systematic bias without eliminating it.
- **Minority-veto for safety:** Any single judge can flag a critical safety issue, blocking the score regardless of majority.
- **Explicit disclaimers in the prompt:** "Do not favor responses based on length. Do not prefer longer answers." Reduces, but does not eliminate, length bias.
- **Fixed decoding parameters:** Temperature=0, consistent sampling reduces variance across judge calls.
- **Calibrate against a human-annotated set:** Run the LLM judge on 10–20 human-labeled cases. Measure agreement rate. Use this as a correction multiplier on judge scores going forward. Without this step, you don't know what your scores mean.

### Building the Evaluation Dataset

- **Golden dataset minimum:** You need 20–50 cases with human-labeled ground truth to calibrate a judge. This is not optional — it's the baseline that makes the entire system interpretable.
- **Data sources for eval cases:** Anonymized production traces (highest fidelity), synthetic data generated by a "dueling LLM" role-playing as a user, human-in-the-loop curation of saved interactive sessions from logs. Production data is gold; synthetic is silver.
- **Dataset versioning:** Pin the eval set to a specific version. Changing the dataset between runs means scores are incomparable. Any score regression or improvement must be attributed to an agent change, not a test set change.

### Connecting Scores to Traces

Every low score must map to a specific trace span. Without trace linkage, a score is a number with no debug path. Tie task success rate to the session trace, step efficiency to the tool call sequence, argument correctness to the specific tool invocation. This is what separates "we measured quality" from "we can improve quality."

## Gotchas

- **Don't run a single LLM judge pass and call it "evaluated."** A single pass has >50% error rate on non-trivial inputs. You need ensemble aggregation, calibration, or both before the score is meaningful.
- **Don't skip calibration.** If your judge says 87% on a task and you've never measured its agreement with human labels on that task type, you don't know what 87% means. Calibrate on 10–20 human-labeled cases first.
- **Don't use LLM-as-judge for safety-critical or domain-expert-dependent assessments.** LLMs lack taste. A judge will confidently score a legally dangerous citation or a subtly incorrect financial calculation as "good" because it looks plausible. Human spot-checks are non-negotiable for these cases.
- **Don't change your eval dataset between runs without versioning.** If you add cases mid-experiment, score changes are uninterpretable. Pin the dataset version, change it deliberately, track the version in your results.
- **Don't ignore deterministic checks.** 60–80% of failure modes can be caught with code graders — schema validation, argument type checks, retrieval precision at k, tool call count bounds. These are fast, free, and reproducible. Start there before paying for LLM judge calls.
