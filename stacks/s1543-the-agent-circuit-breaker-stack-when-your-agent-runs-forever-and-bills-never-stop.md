# S-1543 · The Agent Circuit Breaker Stack — When Your Agent Runs Forever and Bills Never Stop

Your agent goes into production. It calls a broken tool. The tool returns an error. The agent retries. The error persists. The agent retries again — and again. By the time you notice, the run has cost $400 and accomplished nothing. This is not a prompt problem. It is a failure recovery problem, and most teams have no solution for it.

## Forces

- **Agents loop differently than traditional software.** A standard retry loop is built for known-failure shapes (503, timeout). Agents generate their own retry reasoning — sometimes looping for 200 steps because a tool name was misread or a JSON response was malformed.
- **Fail-open is the default in most frameworks.** When an external service crashes, many agent frameworks default to executing the tool anyway. For financial, health, or safety-adjacent agents, this is catastrophic.
- **Token budgets are real money.** Each step in an agent loop costs tokens — input and output. A 100-step runaway run can cost 10–50× a normal run. Unlike traditional software where a loop burns CPU cycles, an agent loop burns your API budget.
- **Failure modes are semantically ambiguous.** Was the action taken? Did the tool execute before the timeout? Did the agent understand the error? Traditional error handling assumes clear failure signals; agent failures are often unclear.

## The Move

Four layered guards — each catching a different failure shape — form the minimum viable failure recovery system for production agents:

- **Hard step cap.** The single most important guardrail. Stop the agent after N reasoning steps regardless of outcome. Recommended default: `MAX_STEPS = 12` for most agents. LangGraph: `recursion_limit=12`. This is the floor, not the ceiling — complex multi-step workflows may need more, but every agent needs a hard ceiling. ([stevekinney.com](https://stevekinney.com/writing/agent-loops))
- **Classify errors before retrying.** Not all errors are retriable. Transient errors (HTTP 5xx, 429 rate limit, connection timeout) → retry with exponential backoff + jitter. Semantic errors (malformed JSON, schema violations, invalid tool call) → re-prompt with corrective context. Resource errors (token budget exceeded, context window overflow) → reduce payload size or abandon. Fatal errors (auth failures, revoked API keys, bad tool schema) → fail immediately, escalate. Retrying non-retriable errors (e.g., a bad API key) creates infinite loops. ([neelmishra.github.io](https://neelmishra.github.io/blog/mlops/llm-agents/agent-error-handling.html), [aiagentsblog.com](https://aiagentsblog.com/blog/agent-error-recovery-patterns))
- **Tool-level circuit breaker.** Track failure rates per tool endpoint. When error rate exceeds a threshold (e.g., >30% failures over the last 60 seconds), open the circuit and short-circuit calls with a structured "dependency unavailable" result. After a cooldown period (e.g., 60 seconds), probe once to check recovery. This prevents the agent from hammering a broken dependency while allowing recovery when it comes back. ([agentpatternscatalog.org](https://www.agentpatternscatalog.org/patterns/circuit-breaker/))
- **Loop detection.** Agents loop in three distinct ways, each requiring different detection. (1) **Exact repetition**: same tool + same args → hash each tool call, detect 3+ consecutive identical calls. (2) **Semantic repetition**: same action with different wording → cosine similarity between embeddings of recent steps, threshold at ~0.85. (3) **Behavioral loops**: plan-execute oscillation, sub-agent recursion, or step-count spiraling → frequency analysis on tool categories. Catch at any layer, halt immediately. ([agent-watchdog/README.md](https://github.com/woodwater2026/agent-watchdog/blob/main/README.md))
- **Cost and budget guards.** Per-run spending cap (e.g., default $1 per task), real-time token counter, alert threshold at 75% of budget. Some teams set a hard kill; others route to human review when a threshold is breached. ([waterwoods.substack.com](https://waterwoods.substack.com/p/im-building-agent-watchdog-heres))
- **Fail-closed for high-stakes tools.** For financial, medical, or administrative tools, enforce deterministic pre-execution checks (Pydantic validation, regex, hard numeric limits) that override LLM reasoning. If validation fails, the action is blocked — regardless of what the agent decided. This is the FailWatch pattern: "Math > Prompts" for hard constraints. ([HN Show HN: FailWatch](https://news.ycombinator.com/item?id=46529092))
- **Escalation hook on terminal failure.** When the agent hits a step cap, circuit breaker, or fatal error: capture full state (all tool calls, outputs, tokens spent), log to observability, surface to human reviewer or ticketing system. The output should include enough context to retry or complete manually. ([rajpoot.dev](https://blog.rajpoot.dev/posts/ai/llm-agent-error-recovery-2026/))

## Gotchas

- **Naive retry loops catch everything.** If your retry decorator catches all exceptions, a bad API key will retry forever. Always classify errors first; retry only transient errors.
- **A step cap without observability is a blindfold.** A step cap stops the runaway but gives you no signal about *why* it ran away. Log every step (tool, args, output, tokens, latency) so you can diagnose the root cause.
- **Loop detection at the string level misses semantic loops.** An agent won't call the exact same function with the exact same args — it'll rephrase. Embed recent steps and check cosine similarity, not just string equality.
- **Cost guards need real-time counting.** A per-run budget checked only at step boundaries can still overshoot on a single expensive tool call. Measure token consumption before each tool call, not just after.
- **Fail-open is the framework default for a reason.** Most frameworks fail open to avoid frustrating users with silent blocks. Enforcing fail-closed for high-stakes tools requires explicit opt-in — don't assume it's there.
