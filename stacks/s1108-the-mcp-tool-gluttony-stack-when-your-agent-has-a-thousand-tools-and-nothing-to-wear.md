# S-1108 · The MCP Tool-Gluttony Stack — When Your Agent Has a Thousand Tools and Nothing to Wear

You wired up an MCP server. Then another. Then you discovered the registry. Now your agent has 47 tools across 8 servers and it spends more tokens deciding which tool to call than actually doing work. The MCP ecosystem exploded to 8,500+ servers in under a year — GitHub (45k deployments), PostgreSQL (32k), Filesystem (28k) — but having more tools does not mean having better agents. The real problems are tool selection overhead, context bloat, permission isolation, and the shift from "give the agent a tool" to "manage a fleet of tool services."

## Forces

- **Context window is the real bottleneck.** Loading all tool definitions upfront into the prompt is the traditional MCP client pattern. With 8,500+ servers and dozens of tools per server, the tool definitions themselves consume a significant fraction of your context budget before the agent does any real work. Anthropic's Nov 2025 engineering post documented this explicitly as a scaling failure mode.
- **Every tool call is an attack surface.** In production, LLM → function call → backend code → tool means the model can trigger anything wired in, with no permission boundary, no isolation between runs, and no audit trail. TengineAI's founder described this as "the wrong abstraction" — LLM triggering application code directly, without an infrastructure layer in between.
- **Discovery is fractured.** MCP servers were scattered across registries, repos, and community threads before GitHub launched its MCP Registry in September 2025. Finding a production-quality server for your use case, vetting it for security, and keeping it updated remains a manual process.
- **Composition is harder than connection.** Wiring up 3 MCP servers is trivial. Making 12 servers work together with shared state, consistent error handling, and coherent permissions across a multi-agent system requires the kind of plumbing that demos never show.

## The Move

The core pattern that separates working agents from production chaos:

- **Treat tools as managed services, not function calls.** Separate the LLM's "I need X" from execution. Use an infrastructure layer (like TengineAI's hosted runtime, or a self-hosted MCP gateway) to enforce permissions, manage credentials, scope API calls, and provide audit logs. The LLM requests; infrastructure runs.
- **Load tools lazily, not upfront.** Anthropic's pattern for MCP code execution uses on-demand tool loading: the agent only receives tool definitions when it actually needs them. This is the key architectural shift from "all tools in context" to "discover tools as needed." Tool filtering (static or dynamic) is now a first-class MCP feature in OpenAI's Agents SDK.
- **Start with the top-3 tool categories, not the full registry.** ToolBoost's deployment data shows the most-used tools are GitHub (45k), PostgreSQL (32k), and Filesystem (28k). For most agent use cases, a code execution runtime, a browser automation layer, and a data retrieval connector cover 80% of real workflows.
- **Use headless terminal as a universal tool adapter.** The ht-mcp project (215 stars, Rust, Apache-2.0) wraps a headless terminal in an MCP server — giving agents a shell they can invoke to run arbitrary commands. It's the most flexible tool because it subsumes every other tool, but it also requires the most isolation discipline.
- **Consider agents-as-MCP-servers for composition.** The lastmile-ai/mcp-agent project (8,424 stars) flipped the paradigm: agents themselves expose MCP interfaces, so an orchestrator can invoke agents the same way it invokes any other tool. This composes naturally without custom protocol code.

## Gotchas

- **Tool explosion does not improve quality.** More tools available to the agent means more decisions about which tool to use, more context consumed by tool descriptions, and more opportunities for the agent to pick the wrong tool. Start minimal; add tools when you have a specific failure mode, not in advance.
- **Security is not an MCP default.** MCP defines the protocol, not the security posture. Every tool connection you add is a potential credential leak, unauthorized action, or privilege escalation. Scoped credentials, permission boundaries, and execution isolation are your responsibility — not the framework's.
- **WebMCP is changing browser automation fundamentally.** Google's WebMCP (early preview) lets sites expose structured tools directly via `navigator.modelContext` rather than agents parsing DOM and simulating clicks. This will make DOM-scrape agents obsolete and require re-evaluating browser automation tool design. — [r/mcp: WebMCP browser-native execution, 2026](https://www.reddit.com/r/mcp/comments/1rdcvkd/webmcp_is_new_browsernative_execution_model_for/)
- **Multi-step tool chains have reasoning overhead.** On r/LocalLLaMA, a production agent developer described the pattern: "LLM 'thinks' between every step. A task that should be 500 tokens of actual work costs 3-4k tokens because of all the intermediate reasoning." Pushing orchestration logic into YAML-defined workflows outside the LLM is one mitigation — keeping the LLM focused on decision points, not procedure. — [r/LocalLLaMA: Multi-step tool chains in production, 2025](https://www.reddit.com/r/LocalLLaMA/comments/1qh8xj6/those_of_you_running_agents_in_productionhow_do/)
