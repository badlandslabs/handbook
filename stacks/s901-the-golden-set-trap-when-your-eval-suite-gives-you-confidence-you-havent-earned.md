# S-901 · The Golden Set Trap — When Your Eval Suite Gives You Confidence You Haven't Earned

Your agent scores 91% on your eval suite. You ship it. Three weeks later it re-onboards the same customer, hallucinating a prior commitment, and nobody caught it because "onboarding correctness" wasn't in the golden set. Your eval suite measures what you thought to test. Agents fail in ways you didn't anticipate — and your benchmark score told you nothing about the failure mode that actually cost you the customer.

## Forces

- **Agents change the eval question.** LLM benchmarks (MMLU, HumanEval) test one-shot answers. Agent benchmarks test sequences: which tools to call, in what order, with what parameters, and how to recover when something breaks. A model scoring 95% on MMLU-Pro can still fail at a multi-step WebArena task requiring synthesis across pages, form entry, and result verification. You are not evaluating a model — you are evaluating a system, and the scaffolding around the model matters more than the model itself.
- **Benchmark scores are scaffolded, not raw.** The same bare model can swing 30–50 percentage points depending on the framework wrapper, prompt structure, tool registry, memory implementation, and retry logic around it. Vendor "92% on benchmark X" claims routinely reflect best-of-16 attempts, contaminated datasets, or custom scaffolding that doesn't transfer to production. Pass@1 (one attempt, production-realistic) and Pass@16 (take the best of 16 runs) are fundamentally different numbers.
- **Aggregate pass rate is a lying average.** "Refund Agent: 88% pass" hides the 12% failure on amounts over $500 caused by a prompt instruction about rounding. The aggregate number looks healthy. The failure mode stays invisible until a customer screenshots the wrong refund amount.
- **Eval inflation is rampant.** 89% of teams with agents in production have some form of observability, but fewer than half have structured evaluations — and most of those evaluate outputs, not trajectories. You can have green dashboards and a broken agent simultaneously.

## The Move

Build a **skill-tree eval system**: decompose your agent into 5–15 discrete skills, write a rubric and regression set per skill, gate merges on per-skill pass-rate regression, and run on every deployment and every model change.

- **Break the trace into skills.** Instead of "Refund Agent: 88%," surface `calculate_refund_amount: 91%`, `verify_customer_eligibility: 87%`, `format_refund_response: 99%`. The skill that regressed tells you where to debug — no days of trace archaeology required.
- **Write 50–100 representative tasks as your golden set.** Cover critical paths, edge cases, and adversarial inputs. Run before every deployment, after every model update, and after every tool or prompt change. This is the single practice that catches most regressions before production.
- **Measure trajectories, not just outcomes.** A correct final answer reached via a broken reasoning path is a failure waiting to recur. Instrument step counts, tool-call sequences, and decision points. Catch the 17% of multi-step failures that produce right answers for wrong reasons.
- **Distinguish benchmark types.** SWE-bench Verified for coding agents. GAIA for real-world multi-turn assistance. WebArena for web automation. OSWorld for OS interaction. Tau-Bench for airline/customer service. No single benchmark covers your agent's actual job — use each for what it's designed to stress-test.
- **Combine LLM-as-judge with human spot-checks.** LLM judges are fast and cheap but hallucinate criticism on non-trivial inputs. Use them for regression speed; use human reviewers for rubric design and edge-case validation. Build a feedback loop where judge disagreements surface novel test cases.
- **Set per-skill pass thresholds, not aggregate thresholds.** If `send_critical_email` drops from 100% to 97%, that 3% regression should block deployment even if the overall score went up.

## Gotchas

- **Your golden set tests what you imagined.** The failure modes that actually break production are the ones you didn't think to test. Complement your golden set with chaos inputs, adversarial edge cases, and production trace analysis to continuously expand coverage.
- **Scaffolding accounts for most of your score.** Before claiming "our agent hits X%," isolate the model change from the framework change. The 40-point improvement you attributed to the new model might have come from switching your tool registry from synchronous to async.
- **LLM-as-judge has a cascading hallucination problem.** The judge LLM can hallucinate both the critique and the original output it's critiquing — producing confident wrong verdicts that look authoritative. Treat judge outputs as a first-pass filter, not a final verdict.
- **Loop detection that counts error types, not iterations, is a budget vulnerability.** If your retry logic increments a counter per distinct error type rather than per total attempt, an agent can issue 47,000 API calls in a single failure mode and your detection never triggers.
