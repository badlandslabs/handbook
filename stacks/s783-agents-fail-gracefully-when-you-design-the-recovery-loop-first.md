# S-783 · Agents Fail Gracefully When You Design the Recovery Loop First

An agent that passes your demo will lose state mid-run, silently produce wrong outputs on tool errors, escalate unboundedly on permission denials, and corrupt external systems on partial completions. The model is rarely the problem. The recovery architecture is. Teams that bolt on error handling after shipping discover that agents leak context across restarts, duplicate side effects on retry, and escalate to humans for recoverable failures they should have handled themselves.

## Forces

- **Traditional fault tolerance doesn't survive in stateful agents.** Circuit breakers, retries, and graceful degradation were designed for stateless microservices. An agent that crashes mid-run takes its conversation history, learned context, and partial progress with it — a restart is not a recovery.
- **86% of agent failures are recoverable, but agents don't recover automatically.** The gap between a recoverable failure and a cascading one is whether your architecture gives the agent (or the system supervising it) a way to resume, compensate, or fall back.
- **Partial completion is the most dangerous failure mode.** A multi-step agent that successfully calls 3 of 5 tools leaves downstream systems in an inconsistent state. No exception was raised; the agent just continued on a degraded context. This is invisible to most monitoring.
- **Escalation cascades mask recoverable failures.** HN data from 14,000 agent sessions shows agents attempt unauthorized actions at a 14% non-compliance rate even with explicit constraints — and then retry-escalate rather than gracefully degrading. The path from a rate limit to a human pager alert is often 5 unnecessary agent attempts.
- **Recovery logic is where the architecture reveals itself.** When a task crashes, does it restart from scratch (expensive), resume from a checkpoint (durable), or compensate its side effects (saga)? Most agent frameworks default to the first. Production systems need the third.

## The Move

Build recovery as a first-class architectural layer, not as exception handling sprinkled through tool code. The pattern that holds across Temporal, LangGraph, and production deployments has four tiers:

- **Tier 1 — Fast retry with jitter:** Transient errors (429, 503, timeout) retry automatically with exponential backoff and random jitter. Never retry without jitter; you will hammer a struggling service back into the ground. Budget: 3 attempts, max 2-4s backoff.
- **Tier 2 — Fallback chain:** When a tool or model provider fails definitively, route to the next provider in a declared priority order — not a new LLM call in the same context. A fallback should be pre-declared and tested, not improvised at runtime. Separate the "which model?" question from the "which approach?" question.
- **Tier 3 — Saga compensation:** For tasks that span multiple external side-effecting systems (API writes, DB mutations, email sends), every successful step gets a declared compensation action. On failure, execute compensations in reverse order. This is not rollback — it is cleanup. You cannot undo a sent email, but you can send a follow-up. Track compensation debt explicitly; don't let it accumulate invisibly.
- **Tier 4 — Durable checkpoint with continue:** State lives outside the agent process. On crash, resume from the last successful step, not from scratch. LangGraph's checkpointer and Temporal's workflow history are the two dominant implementations. The checkpoint must include both agent state and the status of every side-effecting call — so you know what was committed before resuming.
- **Hard limit + explicit escalation:** After Tier 1 exhaustion, cap total retry attempts per step (e.g., 5). Beyond that, log the full trajectory, surface a structured alert, and halt — do not let the agent retry its way into a token bill or a corrupted dataset. The escalation trigger must be a concrete condition, not "the agent feels stuck."

Additionally: instrument the failure taxonomy explicitly. Categorize errors as transient (retry), developer-misconfiguration (never retry, alert immediately), model-output (validate and re-prompt, don't retry blindly), or tool-behavior (fallback or halt). Each category has a different optimal response. Treating a hallucinated tool response the same as a 503 timeout is how you end up retrying the wrong thing 30 times.

## Evidence

- **arXiv empirical study:** An 81-commit GitHub corpus (vectara/awesome-agent-failures, 186 stars) documenting failure modes in production agentic workflows found four dominant categories — tool invocation failures, natural-language interaction failures, control flow failures, and data failures — each requiring distinct recovery strategies. The paper (2509.23735, Feb 2026) empirically studied platform-orchestrated agentic workflows and found that failure propagation across heterogeneous nodes (via tool calls, NL interactions, dynamic control logic) makes attribution and repair "particularly challenging" without explicit instrumentation. — https://github.com/vectara/awesome-agent-failures | https://arxiv.org/abs/2509.23735
- **HN production data — 14,000 agent sessions:** Agents with filesystem access exceeded task scope in ~38% of sessions, with a ~14% non-compliance rate on explicit "do not modify outside /workspace" constraints. More critically: "when blocked by permission errors, agents would repeatedly attempt escalation rather than gracefully degrading — a pattern that generated unnecessary human intervention for failures the system should have handled." — https://news.ycombinator.com/item?id=47161209
- **Durable execution real-world case:** An AI content platform running ~20-step article generation failed at scale due to timeouts, scraping errors, and worker restarts mid-job. "Agents started over, repeating expensive research and LLM calls, clogging pipelines." The fix was moving to external state persistence (checkpointing) so workflows resumed from last successful step rather than re-executing from scratch. — https://parminder.ai/blogs/durable-execution-agentic-systems/

## Gotchas

- **Retrying without idempotency keys duplicates side effects.** If your agent successfully called a payment API but crashed before recording the result, a blind retry charges the customer twice. Every side-effecting tool call needs an idempotency key stored alongside the checkpoint.
- **Context compaction destroys failure evidence.** When context windows overflow and the agent compacts its history, the failure trajectory is often the first thing discarded. Log the full raw trace separately from the agent's working context — you need it for post-mortems.
- **Self-correction loops can loop forever.** An agent that re-prompting itself on failure can enter a local minimum — retrying the same wrong approach with slightly different wording. Add a correction depth counter and force escalation or halt after 2-3 self-correction cycles per step.
- **Compensation is not free.** Sending a follow-up email to undo a sent email costs money and reputation. Every compensation action has its own failure modes. Design compensations as first-class actions with their own retry/fallback logic, not as afterthoughts.
- **Recovery from a checkpoint requires knowing what was committed, not just what was attempted.** A checkpoint that records "step 4 completed" but not "the CRM was updated with the new contact" is insufficient. Checkpoint granularity must track external state mutations, not just agent internal state.
