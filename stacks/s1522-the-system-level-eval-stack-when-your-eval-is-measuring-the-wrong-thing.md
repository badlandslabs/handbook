# S-1522 · The System-Level Eval Stack — When Your Eval Is Measuring the Wrong Thing

You run your agent through its evaluation suite. The model scores 74%. You tune the prompt, switch to a better model, try few-shot examples. The score climbs to 81%. You ship it. Three days in, support tickets pile up about broken integrations, missing API keys, and URLs that return 200 but the wrong body. The eval was measuring the model's quality. The failure was in the plumbing. This is the system-level eval problem: most agent failures in production look like model failures under evaluation but are not model failures at all.

## Forces

- **Eval suites optimize for what they measure, not what breaks.** When you build an eval, you define test inputs and expected outputs. You test what you thought to test. What you did not think to test — a broken URL in a tool call, an API key missing from production, a rate limit that flips the agent into degraded mode — never appears in your golden dataset. The eval passes cleanly while the system breaks silently.
- **Most eval failures are software bugs, not model bugs.** Across multiple teams running agent eval pipelines in 2025–2026, the dominant failure modes surfaced by evaluation were: broken URLs scoring 22, localhost calls in cloud environments scoring 46, real CVEs flagged as hallucinations (eval issue, not model issue), external dependencies returning non-200 responses, and missing environment variables causing silent failure. The agent's quality was fine. The infrastructure around it was not.
- **Agents amplify system fragility across long trajectories.** A single broken link, stale read, or tool returning unexpected output cascades through every subsequent step. The agent looks busy, reasons intelligently, calls the right-looking tools — and produces an output that is confidently wrong in a way no eval caught. By step 8, the root cause (a 404 on step 1) is buried under six layers of plausible-sounding reaasoning.

## The move

The core shift: agent evaluation must be treated as **software testing**, not benchmarking. The agent is a component in a system; the eval must validate the whole system.

- **Build eval suites that capture system-level failures, not just quality failures.** Include test cases for tool unavailability, malformed tool responses, rate limiting, missing credentials, network timeouts, and broken external URLs. These are the failure modes that actually break production, and they are almost never in a standard agent benchmark.
- **Use deterministic checks before reaching for LLM-as-judge.** When evaluating whether a tool was called correctly, whether the right parameters were passed, whether a URL resolves, whether an ID exists in the returned payload — write a deterministic check. It is faster, reproducible, and does not have length bias. Reserve LLM-as-judge for evaluating things that genuinely require semantic judgment.
- **Capture traces, not just scores.** A pass/fail number tells you nothing about why the agent failed. A full execution trace — every tool call, every intermediate state, every LLM reasoning step — tells you exactly which component broke and whether it was a model failure or a system failure. Tools like LangSmith, Langfuse, and Phoenix (Arize) make this tractable.
- **Define success criteria at the system level first.** Before writing a single eval, answer: what does a successful agent run look like from the user's perspective, not the model's perspective? The answer determines what you measure. Most eval efforts start with "what should the model output?" instead of "what should the system accomplish?"
- **Run evals against a staging environment that mirrors production infrastructure, not a mock.** Mocked tool responses let the agent pass the eval while breaking against real infrastructure. Use containerized, version-locked staging environments that simulate real external dependencies, including their failure modes.

- **InfoQ article:** Article on evaluating AI agents in practice argues agents must be evaluated as systems — planning, tool calling, state maintenance, and recovery all matter independently. Recommends hybrid evaluation combining deterministic checks (tool correctness, API response validation), trajectory analysis (was the execution path efficient?), and human review for contextual quality. — [InfoQ](https://www.infoq.com/articles/evaluating-ai-agents-lessons-learned)

Argues that understanding the trajectory (the sequence of reasoning and tool calls) and the full agentic interaction is required for effective debugging, not just grading the final output. — [Google Cloud Blog](https://cloud.google.com/blog/topics/developers-practitioners/a-methodical-approach-to-agent-evaluation)

## Gotchas

- **Defining success by model output, not system outcome, is the most common eval mistake.** If the agent calls the wrong API but produces a plausible-sounding answer, your eval needs to catch the wrong API call — not just grade the answer.
- **Long-horizon agents fail in ways that single-shot evals cannot detect.** An agent can be correct on steps 1–7 and compromised on step 8 due to corrupted state from an upstream tool. Your eval must inspect the trajectory, not just the final output.
- **LLM-as-judge scores do not correlate reliably with production quality.** Judges have length bias, position bias, and self-preference bias. A judge that rates "well-formatted wrong answers" highly will pass regressions that lose data in production.
