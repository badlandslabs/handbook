# S-1502 · The Structured Error Feedback Stack — When You Tell the Agent What Went Wrong So It Can Fix It

Your agent hits a failed tool call and your observability dashboard shows a red spike. You dig in. The model saw a raw Python stack trace 47 lines long in its context, spent 900 tokens reasoning around it, guessed wrong, and continued down a broken path. Meanwhile the actual fix was: "The date field must be ISO 8601 format (YYYY-MM-DD), not MM/DD/YYYY." The agent needed a signal, not noise. Structured error feedback is the difference between an agent that self-corrects and one that self-destructs.

## Forces

- **LLMs can correct errors but cannot find them.** Research from ACL 2024 (Tyen et al., 2024) confirms: state-of-the-art LLMs perform poorly at *locating* their own reasoning errors, but perform well at *correcting* a known error when the location is provided. The implication: your error interface is the bottleneck, not the model's reasoning power. — [https://aclanthology.org/2024.findings-acl.826](https://aclanthology.org/2024.findings-acl.826)
- **Agents plan their own call sequences, breaking standard retry logic.** A tool may be invoked 0, 1, or 3 times depending on model decisions. This makes naive retry dangerous — a failed `send_email` call that retries blindly sends a duplicate. Standard try/catch does not account for this runtime-determined call graph. — [https://agent-works.ai/insights/agent-error-handling-recovery-patterns](https://agent-works.ai/insights/agent-error-handling-recovery-patterns)
- **Raw stack traces are the wrong unit of feedback.** An LLM given "TypeError: cannot read property 'id' of undefined" typically cannot self-correct. An LLM given "Error: user_id field is missing. Expected integer, received null. Provide user_id as a required integer parameter." corrects on the next call in the majority of recoverable cases. The signal-to-noise ratio of error feedback is architectural. — [https://agent-works.ai/insights/agent-error-handling-recovery-patterns](https://agent-works.ai/insights/agent-error-handling-recovery-patterns)
- **Context window budgets are shared between task and error data.** A verbose error dump consumed by the model leaves less room for actual work. Error feedback must be concise and actionable, not a debug artifact. — [https://agent-works.ai/insights/agent-error-handling-recovery-patterns](https://agent-works.ai/insights/agent-error-handling-recovery-patterns)

## The Move

Structured error feedback is a translation layer between failure and context. Instead of dumping raw exceptions into the model's context, you classify the error, extract the actionable signal, and format it as a targeted instruction the model can act on. The goal: zero new LLM reasoning required to understand what went wrong.

- **Classify errors into tiers before deciding what to return.** Separable categories: recoverable tool errors (API timeout, rate limit, invalid input format) — feed back as structured feedback, allow retry; transient model errors (rate limit, malformed output) — apply backoff and retry with the same request; resource errors (token budget, context overflow) — reduce payload and reroute to a smaller model; fatal errors (auth failure, revoked key, policy violation) — abort immediately and escalate. — [https://agent-works.ai/insights/agent-error-handling-recovery-patterns](https://agent-works.ai/insights/agent-error-handling-recovery-patterns)
- **Wrap each tool with an error translator at the boundary, not the tool itself.** Define a mapping from error codes/messages to human-readable, model-actionable strings. Example: `HTTP 422 → "Error: date_format_invalid. Expected YYYY-MM-DD, received '{received}'. Resubmit with corrected date field."` This keeps the tool logic clean and centralizes the feedback contract. — [https://agent-works.ai/insights/agent-error-handling-recovery-patterns](https://agent-works.ai/insights/agent-error-handling-recovery-patterns)
- **Add idempotency keys to every stateful tool.** Before retrying, check whether the operation already completed. Return `"Error: operation already completed (idempotency_key={key}). No action taken."` instead of blindly retrying. This makes retry safe for tools that create records, send messages, or trigger charges. — [https://agent-works.ai/insights/agent-error-handling-recovery-patterns](https://agent-works.ai/insights/agent-error-handling-recovery-patterns)
- **Use per-step time budgets, not a global timeout.** One slow tool (web scrape hitting a CAPTCHA) can consume the entire run budget and starve subsequent steps. Assign a per-call timeout (e.g., 30 seconds) with a clear timeout message: `"Error: tool 'web_fetch' exceeded 30s timeout. The page may be slow or unavailable. Consider using the cached version or skipping this source."` — [https://github.com/ombharatiya/ai-system-design-guide/blob/main/07-agentic-systems/07-error-handling-and-recovery.md](https://github.com/ombharatiya/ai-system-design-guide/blob/main/07-agentic-systems/07-error-handling-and-recovery.md)
- **Pair circuit breakers with model fallbacks.** When a dependency fails repeatedly (third-party API, specific tool), stop routing traffic to it for a cooldown window. Simultaneously route to a fallback: a cached response, a simpler tool, or a cheaper model. A circuit breaker without a fallback just fails faster. — [https://agent-works.ai/insights/agent-error-handling-recovery-patterns](https://agent-works.ai/insights/agent-error-handling-recovery-patterns)
- **Design error handling at tool-definition time, not retroactively.** Retrofitting structured error feedback into existing tools is expensive and error-prone. Treat the error feedback contract as part of the tool schema — alongside input parameters and return type. — [https://agent-works.ai/insights/agent-error-handling-recovery-patterns](https://agent-works.ai/insights/agent-error-handling-recovery-patterns)
- **Distinguish "model output parsing errors" from "tool errors."** When an LLM produces malformed JSON that your parser can't read, the error is in the model's output, not the tool. Return a structured "please retry with valid JSON" message rather than a Python exception. LangChain's `handle_tool_error` flag converts exceptions into LLM-readable observations in the thought/action/observation loop. — [https://engineersofai.substack.com/p/ai-letters-28-agent-error-handling](https://engineersofai.substack.com/p/ai-letters-28-agent-error-handling)

## Evidence

- **Research paper:** LLMs at ACL 2024 (Tyen et al.) tested state-of-the-art models on mistake-finding vs. mistake-correcting. Models correctly identified their own reasoning errors in under 30% of cases but corrected known errors in the majority of cases when the error location was provided. This is the empirical basis for structured feedback: the model is capable, it just needs the error signal delivered correctly. — [https://aclanthology.org/2024.findings-acl.826](https://aclanthology.org/2024.findings-acl.826)
- **HN practitioner thread:** In the HN discussion of Anthropic's "Building Effective AI Agents" (543 points), practitioners with production agents agreed that the biggest lever was not model choice but error signal quality. One contributor noted their agent's retry success rate went from 12% to 67% after switching from raw exception dumps to typed error objects with remediation hints. — [https://news.ycombinator.com/item?id=44301809](https://news.ycombinator.com/item?id=44301809)
- **LangChain production rebuild:** When LangChain rebuilt their documentation chatbot (chat.langchain.com), they switched from vector embedding retrieval to a multi-agent architecture with explicit error boundaries. Each agent subgraph handles its own failure modes with structured feedback loops, which was critical for the two-tier latency requirements (sub-15s for simple queries, 1-3 min for complex analysis). — [https://blog.langchain.com/rebuilding-chat-langchain/](https://blog.langchain.com/rebuilding-chat-langchain/)

## Gotchas

- **LangChain's `handle_tool_error` catches exceptions but doesn't classify them.** Setting `handle_tool_error=True` converts exceptions to LLM observations, but without typed error classes, all failures look the same to the model. Add explicit error type routing so the model gets different instructions for rate limits vs. malformed inputs vs. auth failures.
- **A model that self-corrects once may not self-correct twice on the same error.** After a failed retry, the error feedback loop degrades — the model has already seen the error and failed to fix it once. After two failures, escalate to a human or a different strategy rather than continuing to loop. Implement an explicit "retry count per error" counter.
- **Structured feedback is only as good as the error taxonomy.** If your classification lumps "invalid date" and "API timeout" into the same error type, the feedback will mislead the model. Invest in a precise error taxonomy during tool design — it's the schema of your recovery layer.
