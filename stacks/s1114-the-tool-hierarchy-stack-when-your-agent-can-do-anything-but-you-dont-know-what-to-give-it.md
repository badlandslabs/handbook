# S-1114 · The Tool Hierarchy Stack — When Your Agent Can Do Anything, But You Don't Know What to Give It

You have an agent that can theoretically browse the web, run code, query databases, send emails, and control your desktop. So you give it everything. The agent spends half its tokens deciding which tool to use, hallucinates API calls that don't exist, and still somehow deletes the wrong file. The problem is not the agent. The problem is the tool surface — too broad, too ambiguous, too expensive. This is the tool selection problem, and it is where most agent builds fall apart.

## Forces

- **The cost gradient is extreme.** Computer use (vision-based screenshot analysis) costs ~45× more than a targeted API call for the same task. Agents with full browser access burn budget on tasks a $0.03 function call could solve — not because the model is wasteful, but because nobody scoped the tools correctly. — [DEV Community, 2025](https://dev.to/amitrix/api-connector-browser-computer-use-human-a-cost-justified-hierarchy-for-agent-tooling-119b)
- **More tools do not mean better agents.** Tool overload creates decision paralysis and token overhead. Each additional tool competes for context window space and attention budget. Teams that give agents 20 tools report worse performance than teams with 5 well-scoped ones. — [Anthropic, Building Effective Agents, 2025](https://www.anthropic.com/engineering/building-effective-agents)
- **Tool descriptions are the interface contract.** The agent decides which tool to use based almost entirely on the tool's name and docstring. A vague docstring produces a mis-targeted call. A precise, semantically named tool with a narrow scope produces correct behavior even with weaker models. — [LangChain/LangGraph docs, 2025](https://www.digitalapplied.com/blog/langchain-ai-agents-guide-2025)
- **MCP is collapsing the tool integration surface.** Anthropic's Model Context Protocol is emerging as the de facto standard for agent-tool connection — analogous to USB-C standardizing device connectivity. The alternative is writing and maintaining custom adapters for every integration, which is why teams that adopt MCP report 60-80% reduction in tool-wiring boilerplate. — [deepsense.ai, 2025](https://deepsense.ai/blog/standardizing-ai-agent-integration-how-to-build-scalable-secure-and-maintainable-multi-agent-systems-with-anthropics-mcp/)

## The move

The core technique: **design a narrow, tiered tool surface, not an expansive one.** The best production agents are built around a small set of precise tools, not a large set of capable ones. Here is the hierarchy and the decision logic:

- **Prefer structured APIs over browser automation.** If the data or action you need has a documented API, write a dedicated tool for it. This is $0.03–$0.10 per task vs. $3–5 for computer use on the same task. The agent does not need to understand the page — it needs the answer. — [DEV Community, 2025](https://dev.to/amitrix/api-connector-browser-computer-use-human-a-cost-justified-hierarchy-for-agent-tooling-119b)
- **Write tools that do one thing with a specific output shape.** A tool named `get_customer_orders(customer_id, date_range)` with a typed response schema is better than a tool named `query_database(query)`. Narrow scope → correct tool selection → lower hallucination rate. — [LangChain docs, 2025](https://www.digitalapplied.com/blog/langchain-ai-agents-guide-2025)
- **Use MCP as the integration layer, not custom adapters.** MCP standardizes tool discovery, resource access, and prompt templates across every model and framework. Instead of writing a custom adapter for every new tool, you connect via MCP and the agent can use it immediately. — [deepsense.ai, 2025](https://deepsense.ai/blog/standardizing-ai-agent-integration-how-to-build-scalable-secure-and-maintainable-multi-agent-systems-with-anthropics-mcp/)
- **Sandbox all code execution by default.** Agents that write and run code need isolation — Docker containers, Firecracker microVMs, or services like E2B. The agent workspace should be ephemeral and network-restricted. Production agents that skip sandboxing become attack surfaces. — [awesome-agent-sandboxes, GitHub, 2026](https://github.com/arjan/awesome-agent-sandboxes)
- **Give agents only what they need, not what they could use.** The three tools most production agents actually need: filesystem (read/write scoped to working directory), bash (restricted command set, no `rm -rf`), and web search (with result limit and recency filter). Everything else is context window noise. — [agent-roadmap-2026, GitHub, 2026](https://github.com/codejunkie99/agent-roadmap-2026)
- **Computer use is a last resort, not a first call.** Claude Computer Use and similar vision-based desktop control should be reserved for tasks where no structured interface exists — legacy software, CAPTCHAs, sites that block CDP. For everything else, a Playwright script with a defined output schema is faster, cheaper, and more reliable. — [DEV Community, 2025](https://dev.to/amitrix/api-connector-browser-computer-use-human-a-cost-justified-hierarchy-for-agent-tooling-119b)

## Evidence

- **Benchmarking the cost hierarchy:** Eight runs across browser, desktop, and hybrid configurations showed costs ranging from $0.03 (structured API) to $5.44 (full computer use) for equivalent tasks. The spread came entirely from implementation choices, not the model. — [DEV Community: API Connector vs Browser vs Computer Use, 2025](https://dev.to/amitrix/api-connector-browser-computer-use-human-a-cost-justified-hierarchy-for-agent-tooling-119b)
- **MCP adoption and productivity gains:** Teams adopting Anthropic's MCP reported 60–80% reduction in tool-wiring boilerplate. The protocol standardizes tool schemas, resource templates, and prompt templates across all connected tools, eliminating per-integration custom code. — [deepsense.ai: Standardizing AI Agent Integration with MCP, 2025](https://deepsense.ai/blog/standardizing-ai-agent-integration-how-to-build-scalable-secure-and-maintainable-multi-agent-systems-with-anthropics-mcp/)
- **Code execution sandbox landscape:** The production sandboxing stack for agentic code execution centers on E2B (open-source cloud runtime with Linux OS and SDK), Docker containers for lightweight isolation, and Firecracker microVMs for stronger tenancy isolation. AWS Lambda and Google Cloud Functions are also used. — [awesome-agent-sandboxes, GitHub, 2026](https://github.com/arjan/awesome-agent-sandboxes)
- **HN confirmation:** The Show HN for mcp-agent (80 points, January 2025) validates that MCP is being adopted as the standard tool interface. The author — a former Microsoft engineer who worked on LSP — explicitly frames MCP as "a standardized interface to let AI communicate with software via tool calls, resources and prompts." — [Hacker News: Show HN — Mcp-Agent, 2025](https://news.ycombinator.com/item?id=42867050)

## Gotchas

- **Prompt injection via tool descriptions.** MCP servers that expose tool descriptions to the LLM are vulnerable to tool poisoning — an attacker embeds instructions in a tool's output that manipulate the agent's behavior on subsequent calls. This is not theoretical: 25+ MCP vulnerabilities were documented in 2025 alone, and an NSA advisory was published on MCP security. — [Practical DevSecOps: MCP Security Vulnerabilities, October 2025](https://www.practical-devsecops.com/mcp-security-vulnerabilities); [SecurityWeek: Top 25 MCP Vulnerabilities, September 2025](https://www.securityweek.com/top-25-mcp-vulnerabilities-reveal-how-ai-agents-can-be-exploited/)
- **Tool description drift.** As tools evolve, their docstrings diverge from their actual behavior. A tool that once "returned customer email" now "returns customer email and phone." The agent reads the stale docstring and makes wrong assumptions. Version tool schemas and validate them against behavior. — [LangChain docs on tool creation, 2025](https://www.digitalapplied.com/blog/langchain-ai-agents-guide-2025)
- **Sandbox escape via agent-generated code.** If an agent can write and execute code, and the sandbox is misconfigured, the execution can escape to the host. Firecracker microVMs with minimal device models are the current production standard for preventing this. Docker alone is insufficient for adversarial code. — [awesome-agent-infra, GitHub, 2026](https://github.com/shenli/awesome-agent-infra)
- **Unbounded tool lists inflate context.** Every tool registered with an agent consumes context window for its name, description, and schema — regardless of whether the agent ever calls it. Audit your active tool list quarterly and remove anything not called in 30 days. — [agent-roadmap-2026, 2026](https://github.com/codejunkie99/agent-roadmap-2026)
