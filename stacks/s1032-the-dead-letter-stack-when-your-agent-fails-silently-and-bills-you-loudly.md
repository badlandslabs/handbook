# S-1032 · The Dead Letter Stack: When Your Agent Fails Silently and Bills You Loudly

Your agent ran for 47 minutes, consumed 73,000 tokens, and returned nothing. There was no error message — the model kept calling the same search tool over and over because it couldn't distinguish "no results" from "tool broken." By the time you noticed, the cost was $23. The problem isn't that agents fail; it's that you gave them no way to fail gracefully.

## Forces

- **Retrying at the wrong granularity burns everything.** Retrying the entire agent run replays the full conversation context — pushing you toward token limits — while the failure might live in a single step. Step-level retry is 10–50x cheaper than agent-level retry, but most teams implement it wrong.
- **LLM non-determinism makes classic retry logic dangerous.** A retry of the same prompt can produce a different answer. For operations with side effects (writes, payments, sends), a retry without an idempotency key is a double-execution waiting to happen.
- **Loops look like progress.** The agent is doing *something* — calling tools, generating tokens, moving through steps. Unlike a crash, a loop doesn't surface as an error. It surfaces as a line item on your invoice. The median stuck-loop incident costs $15–30 before human detection at 15 minutes.
- **"Everything is recoverable" is a lie you tell yourself.** Without state checkpointing, a mid-workflow crash means the agent re-starts from scratch on every failure. Without a dead letter queue (DLQ), failed tasks disappear into a void with no audit trail and no human review path.
- **Orchestration failures dominate agent failures.** Anthropic's analysis of 200+ enterprise deployments found 57% of multi-agent project failures originate in orchestration design — not in the individual agents. The wiring between agents is the real failure surface.

## The move

**Structure every agent workflow around three failure primitives: step-level retry with idempotency, stateful checkpointing, and a dead letter queue for unrecoverable failures.**

- **Tag every stateful action with an idempotency key.** Generate a UUID per step execution before the LLM call. Store it alongside the result. On retry, pass the same key — the downstream action checks "have I already done this?" and returns the cached result instead of re-executing. This is the only way to safely retry operations with side effects.
- **Retry at the step level, not the agent level.** Identify the smallest retriable unit. Classify each call: "safe to retry unconditionally" (read-only: embeddings, classification, extraction) vs. "retry with idempotency key" (writes, API calls) vs. "never retry" (payments, sends). This classification prevents the most expensive class of retry bug: re-sending a payment because the first one timed out.
- **Implement a circuit breaker per tool.** Track the failure rate for each tool call over a sliding window (e.g., 10 calls, 5-minute window). When failures exceed a threshold (say, 50%), open the circuit: stop calling the failing tool and return a graceful fallback. Close it only after N successful calls. This prevents retry storms — where a failing tool triggers cascading retries that take down the whole system.
- **Route unrecoverable failures to a DLQ, not a void.** Failed tasks go to a separate queue with full metadata: task ID, failure reason, step number, LLM output, token count, timestamp. Human operators review and either retry with corrections, escalate, or mark as permanently failed. The DLQ is your audit trail and your feedback loop for improving the agent.
- **Detect loops by semantic state, not step count alone.** Track the semantic signature of the agent's state at each step — not just "how many steps" but "what is the agent actually trying to do right now." When the semantic signature repeats across steps, trigger a loop interrupt. Step count alone fails because legitimate long tasks can exceed arbitrary thresholds.
- **Checkpoint state before every tool call.** Use a durable checkpointer (Postgres for production, Redis for lower-stakes workloads). On failure, the agent resumes from the last checkpoint — not from the start. In LangGraph, this is three lines of code. The checkpoint captures the full graph state, including conversation history and tool call outputs.
- **Use a fallback chain for provider failures.** If OpenAI returns a 429 or 500, fall back to your secondary provider (Anthropic, Gemini) rather than failing. The fallback chain should be ordered by cost: try cheapest first, escalate to more capable models only if needed. Rate limit errors should trigger a queue delay, not an immediate fallback.
- **Implement a human-in-the-loop (HITL) gate for high-stakes actions.** Actions with irreversible consequences — refunds, sends, deletes, payments — should pause execution and require human approval. Configure the threshold by risk level, not by whether you remember to add it. A good HITL implementation catches the bad action, generates training data for future automation, and creates an audit log. It is the single highest-leverage safety primitive.

## Gotchas

- **Retrying too aggressively is the default mistake.** Teams set 3–5 retry attempts at the agent level without classifying by failure type. The result: a token limit violation retries 5 times, each time replaying the full context, burning through a massive budget before finally dying.
- **Full-context replay on retry is a silent budget killer.** Every retry that replays the full conversation history adds the entire context cost again. For a 10-step workflow with a failure on step 8, retries replay all 8 steps. Design your checkpointer to support partial replay, not full replay.
- **Circuit breaker thresholds set too high behave as if they don't exist.** A threshold of 90% failures over 100 calls means your system is already in freefall before the breaker opens. Start with 50% over 10 calls and tune from production telemetry.
- **LLM-as-judge for loop detection is expensive and slow.** Some teams use an LLM to determine if the agent is in a loop by comparing current vs. previous state semantically. This adds an LLM call on every agent step — a 2x cost overhead. Semantic loop detection via lightweight state hashing is cheaper and fast enough for most cases.
- **The DLQ is only as good as its review process.** A DLQ that nobody reviews is just a slower void. Treat the DLQ review rate as an SLO — if it exceeds a threshold, escalate the review capacity, not the threshold.
