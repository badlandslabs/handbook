# S-855 · The Multi-Agent Debugging Gap Stack — When Everything Traces but Nothing Explains

You instrumented everything. OpenTelemetry spans across every agent, every tool call, every LLM turn. LangSmith is lighting up with traces. The dashboard shows your 4-agent workflow completing in 2.3 seconds with zero errors. Tuesday a user emails to say the system recommended conflicting actions for three days. Every span was green. The failure was invisible inside the trace. This is the multi-agent debugging gap: you have observability infrastructure but no diagnosis infrastructure. The tools show you *that* the system ran — they don't show you *why* it drifted.

## Forces

- **Traces show activity, not intent violations.** Every span is HTTP 200. The LLM call succeeded. The tool executed. The agent output was syntactically valid. But step 3 chose the wrong tool, step 4 ran with corrupted context from step 3's output, and every subsequent step built on broken assumptions. Standard APM flags crashes — not wrong decisions.
- **Failure cascades hide their origin.** In multi-agent systems, failures propagate backward through handoff boundaries. The agent that surfaces the bad output is rarely the one that generated the bad input. You need causal tracing, not just connected spans.
- **88% of agent failures are infrastructure, not models.** Across 591 documented production incidents, the dominant cause was missing guardrails, absent monitoring, and inadequate trace instrumentation — not model quality. The fix isn't swapping the LLM. It's adding the diagnostic layer.
- **The observability-to-evals gap is structural.** 89% of teams running multi-agent systems have tracing. Only 52% have evals. Teams can see what happened but can't answer "was this right?" — so they rebuild and cross their fingers.

## The Move

The move is layered: add causal tracing that captures decision rationale (not just inputs/outputs), instrument the handoff schema layer specifically (not just the agent internals), and close the loop by converting every production failure into a regression test. Three concrete layers:

- **Capture "why" at every handoff, not just "what."** Every agent-to-agent boundary should log the routing decision and the receiving agent's first-reasoning output, not just the message payload. When a failure surfaces, you need to replay the routing choice, not just the payload. Tools like `agent-triage` (Converra) extract behavioral rules from system prompts, replay each conversation step-by-step with an LLM-as-judge, and flag which turn first broke — across routing, handoffs, and retrieval. The key signal: "24 out of 51 failures are missing escalations" — aggregated root causes, not individual traces.
- **Schema-version every handoff contract.** Untyped handoffs kill multi-agent workflows faster than any other issue. At every agent boundary, enforce a validated schema with version numbering. When the schema changes, old agents fail loudly instead of silently misinterpreting the new payload. This is the contract layer that makes cascading failures attributable.
- **Convert every production incident into a regression case within 24 hours.** Trace → cluster → attribute → test. The Zylos Research trace-driven debugging workflow: cluster similar failures, trace backward to the first divergence from expected path, attribute to a specific step (often a tool returning an empty response the agent interpreted as "no results" vs. "query truncated"), and write a regression test before the sprint ends. This closes the 40%-of-sprint-time investigation tax.

## Gotchas

- **LLM-native observability ≠ agent-native observability.** LLM-native platforms log independent events that require manual correlation. Agent-native platforms (LangSmith, Phoenix) capture causal dependencies between steps. For multi-step workflows, this difference is night and day — you need parent-child span relationships, not just event logs.
- **Tool failures with HTTP 200 are the silent killer.** The most common misdiagnosis in multi-agent debugging is blaming the model when the real problem is a downstream tool returning a valid but empty/wrong response that the agent treated as correct. Instrument tool outputs at the schema level, not just the call level.
- **Retroactive diagnosis is expensive and error-prone.** Trying to reconstruct "what went wrong" from raw traces after the fact takes 3–5x longer than diagnosing during execution. The fix isn't better tracing — it's writing the regression test within 24 hours of any production incident while the trace context is fresh.
