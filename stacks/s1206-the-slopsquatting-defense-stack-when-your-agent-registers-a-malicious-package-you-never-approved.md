# S-1206 · The Slopsquatting Defense Stack — When Your Agent Registers a Malicious Package You Never Approved

Your AI coding agent generates a library name that doesn't exist, suggests installing it, and — in auto-install mode — silently executes a postinstall script from a package an attacker registered against that exact name. 4,000 developers got OpenClaw this way in February 2026. No exploit was staged. The vulnerability was the agent's confidence.

## Forces

- **Agents become package maintainers.** When an LLM recommends a library, it creates a naming dependency no human reviewed. In auto-install mode, the agent is the maintainer — it picks the name, triggers the install, and runs whatever postinstall hook exists. That is not a configuration mistake; it is an architectural inversion of the trust model.
- **Hallucinated names are reproducible.** Researchers tested five models across 576,000 code generation samples and found 127 package names hallucinated identically by all five models. 43% of hallucinated names appear on every repeated run. An attacker doesn't need to guess — they can test the name, find it missing, register it, and wait.
- **The agent has your credentials.** A coding agent running in your CI environment or on your developer workstation has access to environment variables, secrets, internal package registries, and network access. Code execution in this context is not a sandbox escape — it is credentialed access to everything the agent can see.
- **Traditional supply chain scanning misses the trigger.** Your Snyk/Dependabot scan covers packages you chose. It has no opinion about packages your agent chose mid-task, because those packages didn't exist when you wrote the lockfile.

## The Move

### The Slopsquatting Attack Anatomy

The attack requires three conditions to align:

1. An LLM generates a package name from a hallucination (not a typo — a confident but non-existent library name)
2. An attacker monitors for unregistered names, registers the name with malicious code, and waits
3. A downstream agent or pipeline installs the package (typically via auto-install or YOLO mode)

The February 17, 2026 OpenClaw incident had all three: an LLM hallucinated a package name, attackers registered it with a malicious postinstall script, and 4,000 Cline users received the payload silently during an update. The eight-hour window between installation and detection was enough for credential exfiltration.

Research from Seth Larson (Python Software Foundation) found 127 names hallucinated identically across all tested models — meaning an attacker can build a pre-registration list from a single prompt. USENIX Security 2025 data shows 21.7% hallucination rate on open-source models, 5.2% on commercial — but both hit the same names consistently.

    """Return triage decision for an agent-suggested package name."""

    result = {
        "name": name,
        "registered": False,
        "collision": False,
        "intent_match": True,
        "action": "install",
        "warnings": [],

    # 1. Collision check — is the name already registered?
    check = subprocess.run(
        ["pip", "index", "versions", name],
        capture_output=True, text=True
    )
    if check.returncode == 0:
        result["registered"] = True
        result["collision"] = True
        result["warnings"].append("name already registered — verify author/publisher")
    else:
        # Name is hallucinated — high risk
        result["intent_match"] = False
        result["warnings"].append("package does not exist — hallucinated name")

        if auto_install:
            result["action"] = "BLOCK"
            result["warnings"].append("AUTO-INSTALL BLOCKED for hallucinated package")

            result["action"] = "review_required"

### Defense Layers Beyond the Triage

**Layer 1 — Pre-registration watchlist.** Run candidate package names through a known-hallucination list before installation. Seth Larson's research produced a reproducible set of 127 cross-model hallucinations. Maintain this as a deny-list updated monthly from model test runs.

**Layer 2 — Private package registry with pinned snapshots.** Rather than reaching public PyPI/npm at install time, mirror approved packages into an internal registry (Artifactory, Verdaccio, GitHub Packages). Agent installations resolve against the internal registry. Names not in the registry go to a manual approval queue.

**Layer 3 — No auto-install for agent-generated names.** Separate the generation context (where the model suggests packages) from the installation context (where packages are actually installed). Never pass agent output directly to a package manager without an explicit human or policy review step in between.

**Layer 4 — Secrets scoping for agent environments.** Even when slopsquatting succeeds, limit blast radius. Agent environments should run with minimal credentials — no long-lived tokens, no access to production secrets, no npm/pip tokens with write access. Use workload identity (GCP Workload Identity, AWS IAM Roles) to scope access to only what the task requires.

**Layer 5 — SBOM drift detection.** Generate an SBOM before every agent-run installation. Compare against the baseline dependency graph. New packages, unexpected transitive dependencies, or version mismatches trigger a security review gate before the install is committed to the dependency graph.
