# S-1385 · The Agent Tool Ecosystem Stack — When Your Agent Reaches for a Tool That Isn't There

You've built an agent that can reason beautifully about your codebase, your data, your users. Then someone asks it to check the production dashboard. It can't. It has no browser access. Someone asks it to update the Jira ticket. It can't. It has no API integration. The gap between what your agent *understands* and what it *can act on* is where most agent deployments stall — and the solution isn't adding more prompt instructions, it's building the right tool surface.

## Forces

- **The tool surface defines the agent's ceiling.** An agent with only text I/O can analyze, summarize, and draft. An agent with filesystem, browser, API, and code execution tools can *do*. The difference between a useful agent and a useless one is often the difference between three tools and thirty.
- **MCP is winning the tool-connection layer, but tool *selection* is still unsolved.** The Model Context Protocol has become the USB port for AI agents — 150+ integrations in the LangChain community alone, MCP servers for GitHub, Slack, Google Drive, Jira, Postgres, and more. But having 40 MCP servers loaded simultaneously creates a new problem: the agent doesn't know which tool to reach for, or reaches for the wrong one with high confidence.
- **Token economics force tool design discipline.** A single MCP server with 35 tools (e.g., GitHub) costs ~26K tokens just in tool definitions before the conversation starts (Anthropic, 2025). Anthropic's Tool Search Tool solves this by letting the agent *discover* tools on-demand rather than loading all definitions upfront — a pattern that forces you to think about which tools genuinely belong in the core prompt vs. the on-demand library.
- **Sandboxing is non-negotiable for code execution.** E2B, the open-source sandboxed execution environment, is embedded in production stacks at Compound AI, Gumloop, and hundreds of other teams. The security surface of giving an LLM shell access — even a contained one — demands real sandboxing with filesystem boundaries, network restrictions, and process limits. Microsoft research documented real-world spear phishing attacks in early 2025 where AI agents with tool access were actively exploited in multi-turn conversations.
- **Browser automation is the gap most teams forget.** "The future is agents with computer use" — Anthropic, Oct 2024 — but most production agents still lack browser access. Browser Use (91k+ GitHub stars as of 2026, 89% success on WebVoyager benchmark) has become the community default for web automation. The alternative — native Computer Use from Anthropic/Claude, OpenAI CUA/Operator, or Gemini DOM-awareness — each targets different surfaces (containerized browser, desktop GUI, structured web forms).

## The Move

Build a **tiered tool surface** with three layers, not one sprawling tool list:

- **Core tools (always loaded):** Filesystem read/write, shell/Bash, web search, and 1–3 domain-specific APIs that define the agent's primary job. Keep this under 10 tools. Token costs compound fast at scale.
- **MCP-discovered tools (on-demand):** Use Anthropic's Tool Search Tool pattern (or equivalent) to let the agent discover and use MCP servers only when needed. Don't load 40 servers upfront. Load the top 5 for the task class and let the agent query the rest.
- **Privileged tools (escalation only):** Code execution, payment APIs, write/delete operations, external credential use. These require explicit confirmation steps, tighter sandboxing, or human-in-the-loop gates. Never give these to the base agent loop.

**For browser automation specifically:** If your agent needs to interact with SaaS tools, internal dashboards, or sites without APIs, use the MCP + Playwright approach (lowest setup friction, ~$0.002/call) for structured tasks, and Browser Use for autonomous multi-step web navigation. Reserve native Computer Use (Claude/Operator/Gemini) for cases where no MCP exists and the task requires general-purpose GUI navigation.

**For code execution:** Use a hosted sandbox (E2B, Modal, or AWS Lambda) rather than raw shell access. E2B specifically has production deployment patterns documented by Compound AI and Gumloop — Gumloop's Head of Engineering reported: *"LLM-generated API integrations to external services make Gumloop incredibly useful. E2B is essential to make that happen at scale and deliver reliable performance."*

## Gotchas

- **Don't load all tools at startup.** Teams add tools during development, end up with 30+ tools in the prompt, watch token costs balloon, and then can't figure out why their agent is slow and expensive. The MCP discovery pattern (load 5, discover the rest) is the correct response to tool proliferation — not adding more tools.
- **Browser automation is brittle without fallback.** Websites change their DOM structure constantly. A Browser Use agent that navigates LinkedIn today may fail on the same task next month. Build retry logic with visual fallback (re-screenshot and re-parse) and explicit step-count limits to prevent infinite loops on structural changes.
- **Code execution without sandboxing is a security incident waiting to happen.** Even in a "read-only" context, LLM-generated code can escape intended boundaries. Microsoft documented active exploitation of agent tool access in spear phishing campaigns in early 2025. Use E2B, Modal, or Lambda for any code execution that touches network or filesystem.
- **Tool schemas drift.** An MCP server updates, the tool's input schema changes, and your agent starts failing silently with schema mismatch errors. Pin MCP server versions and add schema validation in your integration tests — don't let tool schema drift be a production incident.
- **The "tool call" is not the action.** An agent that *successfully* calls a browser navigation tool has not *completed* the task — it has only issued a command. The agent must observe the result (re-screenshot, check the DOM state) and reason about whether the action succeeded before proceeding. Most "agent stuck in loop" issues are failures to observe, not failures to act.
