# S-1313 · The Failure Barrier Stack — When Your Agent Returns 200 OK and Everything Is Wrong

Your agent completes a 4-hour workflow and logs success. Three days later you discover it wrote duplicate CRM records, sent emails to wrong addresses, and billed two clients for the same task. HTTP 200 on every call. No exceptions. The monitoring dashboard showed green. This is the core problem of agent failure handling: the worst failures look like successes, and most teams have no barrier between a confident wrong answer and a destructive action.

## Forces

- **Agents fail non-deterministically.** A conventional service crashes and logs a stack trace. An agent may silently loop for 35 minutes, spawn redundant subprocesses, accumulate context until the model halts, or take an irreversible action before a human can intervene. Traditional error handling is necessary but structurally insufficient.
- **Pipelines compound failure.** A 10-step pipeline where each step has 85% reliability achieves only ~20% end-to-end success. Multi-agent systems fail at 41–86.7% rates due to spec ambiguity and coordination breakdowns. At scale, failure is not an edge case — it is the expected operating condition.
- **Observability tools are flight recorders, not prevention systems.** LangSmith, LangFuse, Arize, and Helicone show what happened after the fact. They do not prevent collisions. LangFuse's alerting feature request was open for over two years before being implemented (GitHub issue langfuse/langfuse#714, discussed on DEV Community, March 2026). Loop detection, silent degradation alerts, and cost anomaly detection are structural gaps across the industry.
- **The LLM cannot reliably judge its own reasoning.** Intrinsic self-correction — the model critiquing itself — is fragile. Grounded self-correction, anchored in execution results or structured external validators, is what actually works in production.

## The Move

Layer five independent failure barriers. None alone is sufficient. Together they transform an unreliable autonomous system into a survivable one.

- **Hard loop prevention — the outer shell.** Set explicit per-run limits on LLM calls and tool invocations (e.g., `ModelCallLimitMiddleware`, `ToolCallLimitMiddleware`). This is the floor, not the ceiling. LangChain's Deep Agents docs classify excessive calls as a system-level problem solved by caps, not by hoping the model notices it is stuck.
- **Structured tool result schemas — eliminate ambiguous completion.** Make every tool return an explicit `status` field: `complete`, `partial`, `failed`, with a `message` that a model can reason about. Empty arrays and `null` values are not completion signals — they are ambiguous. AIWave's analysis of ReAct pattern failures (Hashnode, July 2026) found this single change eliminates roughly 40% of production loops. A search returning no results should say: `{"status": "complete", "result": [], "message": "Search completed. No results found. Do not retry."}`
- **Validation gates — catch hallucination before execution.** Validation gates sit between tool execution and downstream effects. They validate outputs against schemas, check business logic constraints, and halt execution when outputs are implausible. ValueStreamAI's 2026 benchmarks estimate ~70% of hallucinated outputs are caught before tool execution with structured validation. This is the highest-leverage intervention for preventing silent data corruption.
- **Grounded self-correction — retry with evidence, not self-judgment.** When a step fails, feed the execution result back to the model with a concrete error signal: what happened, what the constraints are, and what changed since the last attempt. Do not ask the model to judge its own reasoning — ask it to revise based on new information. The Zylos Research taxonomy (May 2026) distinguishes "LLM-recoverable errors" (tool failures, parsing issues) from programming bugs, applying self-correction only to the former. LangChain's fault tolerance guide categorizes errors by who fixes them: system (transient), LLM (recoverable), human (missing info).
- **Stateful rollback and recovery — make failure cheap.** For long-running workflows, checkpoint state at each major step. When a step fails, resume from the last checkpoint rather than restarting from scratch. LangGraph's `checkpointer` and Microsoft Agent Framework's checkpoint/resume primitives exist for this. This turns expensive failures into cheap interruptions.
- **Circuit breaker with policy fail mode.** A breaker that trips on repeated failures and blocks subsequent calls is table stakes. The missing piece is an explicit policy for what happens to in-flight actions when the breaker opens. Cordum's 2026 analysis recommends pairing breaker state with explicit closed/open behavior: retry-only amplifies outages at scale (500 jobs/min with 3 retries adds 15,000 avoidable calls over 10 minutes), while a classic breaker with no policy leaves autonomous actions in an undefined state.

## Gotchas

- **Setting a loop limit is not the same as handling failure.** Teams add a max-turns cap and call it done. The agent hits the cap and returns a partial result with no error. Downstream systems treat partial success as success. The cap buys time; you still need a recovery path.
- **Validation gates must run before effects, not after.** If you validate output after writing to the database, you have already corrupted data. Gate placement is a causal question, not a code organization question.
- **Self-correction does not work on reasoning errors.** Reflexion-style self-critique works for execution failures (tool didn't work, parsing failed). It fails on reasoning errors because the model cannot reliably detect flawed logic with the same model. For reasoning errors, use a separate validator model, execution-based verification, or a process reward model (PRM).
- **Checkpointing adds complexity that bites you on recovery.** If your checkpoint schema changes between deploys, resuming from an old checkpoint can corrupt state. Treat checkpoints as versioned, not just persisted. Validate checkpoint compatibility before resuming.
- **Cost limits are a failure mode, not a solution.** Capping spend prevents runaway bills, but an agent hitting its budget limit and stopping mid-workflow is still a failure — just a silent, cheaper one. Budget limits are a floor, not a strategy.
