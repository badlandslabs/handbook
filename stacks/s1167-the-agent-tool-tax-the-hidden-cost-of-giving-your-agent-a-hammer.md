# S-1167 · The Agent Tool Tax — The Hidden Cost of Giving Your Agent a Hammer

Agents need tools to operate in the real world. Every tool you give an agent unlocks a class of capability — but also adds token overhead, latency, error surface, and security exposure. The agent tool tax is the compound cost of tooling decisions: too few tools and your agent is blind and powerless; too many and it drowns in choice, burns tokens on tool selection, and surfaces cryptic failures from tools it barely understands. The MCP ecosystem has made it trivially easy to give agents thousands of tools. Nobody has made it easy to know which ones are worth giving.

## Forces

- **Tool description bloat collapses context.** Loading 50 MCP tool definitions into a prompt consumes 50,000+ tokens before the agent even sees the user request. Anthropic measured this directly on their advanced tool use engineering blog — the tool tax is not metaphorical, it is measured in context window and dollars.
- **Specialized beats general every time.** A direct API call tool (GitHub REST, Salesforce API) outperforms a general browser-use agent on that domain. But you don't know which domains your agent will encounter until production tells you.
- **Sandboxing is the hard problem, not tool discovery.** MCP solved the "how do I connect tools to agents" problem. It did not solve "what happens when this tool does something destructive." Every filesystem write, every API call, every code execution is a potential blast radius.
- **The tool ecosystem exploded faster than the tooling for tool governance.** MCP went from 100K to 97 million monthly downloads in 18 months. Over 1,000 MCP servers launched. The supply of tools vastly outpaced the frameworks for deciding which tools to grant, version, audit, and revoke.

## The move

**Audit tool additions with a three-axis cost model before wiring them up:**

- **Token cost:** How many tokens does this tool's schema add to every prompt? MCP's `Tool Search Tool` (Anthropic, Nov 2025) achieves 85% token reduction on tool loading — use it before hand-wiring every tool definition into the system prompt.
- **Failure surface cost:** What is the blast radius if this tool returns corrupted data, times out, or behaves unexpectedly? Browser-use agents fail in ways filesystem agents don't; code execution agents fail in ways web search doesn't. Size the blast radius before granting access.
- **Isolation cost:** Is this tool running in a sandbox, or does it have direct access to your infrastructure? Terminal Use (YC W26) ships with containerized filesystem access. Tilde.run wraps the filesystem in lakeFS transactions so any destructive agent action is rollback-able. Tools without isolation are liabilities.

**Wire MCP as your tool protocol, not your tool catalog.** MCP solved the integration problem — it is the USB-C of agent toolchains. But the standard's breadth is a trap: do not wire every MCP server you find into your agent. Treat MCP as a protocol layer and maintain a curated, versioned allowlist of tools your agent is permitted to invoke.

**Give agents a tool taxonomy, not a tool list.** Anthropic's "Programmatic Tool Calling" (Nov 2025) enables agents to invoke tools programmatically rather than describing every parameter in the prompt — reducing inference overhead 37% and cutting latency. Structure your tool registry with categories (filesystem, API, computation, web) and grant access by category. An agent that knows "I have API tools, filesystem tools, and computation tools" can self-route better than one handed 47 flat tool definitions.

**Separate tool discovery from tool execution.** Anthropic's Tool Search Tool lets the model search a tool registry at runtime rather than loading all tool definitions upfront. If you build a custom agent, implement a tool discovery step before tool execution — let the agent figure out which tool category is relevant, then invoke the specific tool. This is the architectural pattern that unlocks Anthropic's 85% token reduction.

**For browser access: deterministic state over screenshot.** The dominant failure mode in browser-agent systems is stale page state — the model acts on what it last saw, not what's currently on screen. The Agent Browser Protocol (theredsix/abp, HN Apr 2025) forks Chromium to keep the acting agent synchronized with browser state at every step. browser-use (77K+ GitHub stars) takes a different approach — built on Playwright, it abstracts the browser behind a natural-language interface. Pick your tolerance: ABP for deterministic precision, browser-use for developer ergonomics.

**For code execution: always sandboxed, always ephemeral.** Claude's code execution tool runs Python and bash in a sandboxed container. TakoVM (Tako-Research) wraps code execution in a secure VM with filesystem isolation. If you're building custom code execution, the minimum bar is an ephemeral container with no network access and a hard timeout. Never give an agent code execution in the host environment.

## Gotchas

- **Do not wire every MCP server you find.** The ecosystem has 1,000+ MCP servers. Wire all of them into your agent and you'll spend more tokens on tool definitions than on the actual task. Curate your tool allowlist like you curate your npm dependencies.
- **Browser automation tools fail silently on anti-bot defenses.** browser-use and computer-use agents will attempt actions on sites that block them, return captchas as page content, or silently alter behavior under automation detection. Build checks for the absence of expected content before assuming a browser action succeeded.
- **Tool version drift breaks agents in production.** An MCP server you tested in March may have changed its parameter schema by April. Pin tool versions, run integration tests against the pinned version, and treat tool changes as breaking changes to your agent.
- **Sandbox escapes are real.** Code execution, filesystem access, and browser control are powerful primitives that double as privilege escalation vectors. Tilde.run's transactional filesystem and Terminal Use's containerized approach are not paranoid over-engineering — they reflect the actual blast radius when a misbehaving agent gets an unintended capability.
- **The Anthropic token reduction numbers (85%, 37%) apply to their specific implementation.** The principle — defer tool discovery to runtime rather than loading all definitions upfront — transfers, but the specific percentages are from Anthropic's proprietary implementation of Tool Search and Programmatic Tool Calling. Treat them as directional, not guaranteed.
