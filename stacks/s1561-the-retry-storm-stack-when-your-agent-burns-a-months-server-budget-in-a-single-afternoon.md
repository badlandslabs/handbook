# S-1561 · The Retry Storm Stack — When Your Agent Burns a Month's Server Budget in a Single Afternoon

Your agent's LLM calls succeed. Every single one returns 200 OK. But the downstream operation fails — a column doesn't exist, a rate limit fires, an auth token expired. The retry logic kicks in, and now you've re-sent the entire conversation context 47 times before someone notices. One day of AI cost more than the entire month of server infrastructure. The failure wasn't that the agent failed — it was that the agent kept succeeding at the wrong thing. The Retry Storm Stack is the layered defense architecture that prevents a single flaky API call from cascading into a full token-budget meltdown.

## Forces

- **Retries are not free in agentic systems.** A traditional microservice retry resends an HTTP request — a few KB. An agent retry re-sends the entire conversation context to the LLM. Ten retries don't cost 10× one request; they cost 10× the full context token budget. Uncontrolled loops can produce **200x token cost** relative to a single successful execution.
- **Agent failures are not transient.** Standard retry logic assumes failures reset between attempts — network timeouts, temporary outages. When an agent chooses the wrong tool, misreads intent, or acts on stale context, retrying doesn't reset the problem. It amplifies cost and confidence in the wrong answer. The agent will retry into the same wrong answer every time until something external stops it.
- **Production tool call failure rates dwarf what benchmarks assume.** Industry data shows **12–18% tool call failure rates** in production agent pipelines. At a 5% per-call error rate across 10 steps, a workflow has only 60% probability of completing without retry logic. Tool reliability issues — rate limits, auth rot, schema drift, partially failed responses — are the top failure mode that benchmarks like SWE-bench never test.

## The Move

A layered defense stack — not a single pattern, but four gates working together:

1. **Hard step budgets as the first line.** Cap the maximum number of agent loop iterations at the orchestrator level (10–50 steps depending on task complexity). This is a ceiling, not a suggestion. When the budget exhausts, the agent stops, logs the partial result, and escalates — regardless of whether the task appears "almost done."

2. **Conversation-level token budgets.** Track cumulative token spend per session, not just per call. Set a cost ceiling that halts the entire run, not individual steps. This prevents the slow accumulation of cost from a long-reasoning agent that isn't failing loudly.

3. **Circuit breakers on tool calls, not just API endpoints.** A tool circuit breaker trips after N consecutive failures on a specific tool (e.g., 3 failures calling the database MCP server) and stops routing requests to that tool for a cooldown period. Different from infrastructure circuit breakers — this is a *reasoning-layer* breaker that prevents the agent from repeatedly calling a tool it keeps failing against.

4. **Failure-mode-aware retry logic.** Classify failures into three buckets: (a) **Transient** — timeout, rate limit, 503 — retry with exponential backoff (1s, 2s, 4s, 8s); (b) **Schema/interface** — bad response shape, missing field — do not retry the LLM call, fix the schema or use a fallback; (c) **Permanent** — capability gap, permission denied, tool doesn't exist — escalate immediately, do not retry. Most teams treat all three the same and wonder why retries explode their costs.

5. **Honest degradation on repeated failure.** After circuit breakers trip and budgets exhaust, degrade gracefully: return the last known good state, surface a structured "I couldn't complete this" response with partial results, and route to human escalation. Never silently return empty responses.

6. **Checkpoint before high-risk operations.** Before any tool call that modifies state (write to DB, send email, deploy code), write a checkpoint. If the operation fails and retry would re-apply it, idempotency keys prevent double-application. This is what turns "retry success" from a billing disaster into a normal operation.

## Gotchas

- **The agent succeeded, the operation failed.** The most dangerous scenario: every LLM call returns 200, but the business operation (DB write, API call, file write) fails. Your monitoring sees "no errors." Your cost bill sees a disaster. Instrument at the *operation* level, not just the LLM API level.
- **Idempotency keys are not optional.** Without them, retry-on-success (the LLM call succeeds, but the downstream write fails, triggering a retry that re-runs the LLM call and then successfully writes) can double-apply side effects. "Success" that re-bills a customer is not a recoverable failure — it's a data integrity incident.
- **Step budgets catch runaway, not drift.** A hard step cap stops the agent from looping forever, but it doesn't stop the agent from spending 40 expensive steps on a task that was actually solvable in 5. Combine budgets with progress metrics: track whether each step meaningfully advances toward the goal, and abort if the last N steps produced no progress.
- **Do not retry schema failures.** The agent will hallucinate the same tool arguments against the same schema the same way every time. Classify schema errors as non-retryable, surface the error to the developer (not the agent), and fix the tool definition.
