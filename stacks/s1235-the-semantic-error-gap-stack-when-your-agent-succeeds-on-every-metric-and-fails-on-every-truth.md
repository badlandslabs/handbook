# S-1235 · The Semantic Error Gap Stack — When Your Agent Succeeds on Every Metric and Fails on Every Truth

When the API returns 200, the JSON is valid, the tool executed cleanly, the agent declared victory — and the output is completely wrong. Traditional error handling catches crashes and exceptions. It does not catch confident nonsense, a hallucinated CRM entry that looks perfectly plausible, or a reasoning chain that arrives at the right conclusion via the wrong path. You need a second signal to tell you whether the output is actually correct, not just whether it executed without throwing.

## Forces

- **Execution success ≠ output correctness.** Agents can fail silently at the semantic layer — returning HTTP 200, valid JSON, a confident explanation — while producing factually wrong results. No try/catch, no timeout, no circuit breaker catches this. It is the failure mode that has no traditional software analog.
- **LLMs cannot reliably self-verify at the quality bar production requires.** Self-correction research (Tyton/Bhagoji et al., 2024) shows LLMs can fix some errors but also confidently double down on wrong answers. Intrinsic self-correction is not a production-grade validation strategy on its own.
- **Judge LLMs add latency and cost but eliminate an entire failure category.** A second LLM pass to verify the first adds 50–200ms and doubles token spend — but it catches hallucinations, wrong tool interpretations, and reasoning errors that no deterministic check can detect.
- **Threshold calibration is the hard part.** A judge that flags too much triggers unnecessary escalations. One that flags too little passes bad outputs through. Getting the confidence threshold right requires domain-specific calibration that most teams skip.
- **Structured output validation catches syntax, not semantics.** Pydantic schemas and JSON mode guarantee valid JSON — they do not guarantee that the JSON contains the right answer, that the agent interpreted the user's intent correctly, or that the selected tool was the appropriate one.

## The move

**Install a semantic validation layer between the agent and the outside world.** The agent produces an output or takes an action; a validation gate evaluates it for correctness, consistency, and safety before it is committed. This is not retry logic — it is a separate judgment call from a different model or prompt.

- **Structured output with semantic guardrails.** Use Pydantic models to enforce schema correctness, but wrap each field with a verification prompt that asks: "Given the original user request and the retrieved context, is this field value plausible?" For high-stakes fields (IDs, amounts, addresses), add deterministic cross-checks that don't require a model.
- **Judge-based validation gates.** Route agent outputs through a second LLM call with a structured prompt that evaluates correctness against the task, consistency with prior steps, and flags for escalation. The judge should have access to the original task context, tool results, and a rubric of acceptable vs. non-acceptable outputs. More than 50% of surveyed production agent teams now use judge LLMs at runtime (Zylos Research, 2026).
- **Confidence thresholding with escalation paths.** Configure the judge to emit a confidence score (0–1) alongside its verdict. Route outputs above threshold to the next step; route outputs below threshold down one of three paths: re-prompt with specific critique, surface to human reviewer, or halt with a structured error. The threshold should be tuned per-task — customer-facing summarization tolerates lower confidence than financial transactions.
- **Self-consistency sampling.** Run the same agent step 3–5 times with temperature variation and compare outputs. Outputs that diverge across runs are flagged as uncertain. This is cheap relative to a full judge pass and catches reasoning instability early.
- **Provenance tracking on every fact.** Tag each output claim with its source (tool result, retrieved document, model knowledge cutoff). If an agent produces a claim with no traceable source, surface it to the judge as a hallucination candidate. Tools like LangSmith and Phoenix can log these traces automatically.
- **Idempotency gates on side-effect operations.** Before executing any write operation (email send, database write, API POST), check whether this specific operation has already been attempted with the same parameters. Idempotency keys prevent duplicate sends from retry loops — the leading cause of "graceful degradation that went ungraceful" in production agent incidents.

## Gotchas

- **The judge model can hallucinate too.** A judge LLM is not ground truth — it can be confident and wrong. Calibrate judge thresholds against a golden dataset of known-good and known-bad agent outputs. Without calibration, you're substituting one model's confidence for another's.
- **Validation gates create latency.** A judge pass adds 50–200ms; self-consistency (3 runs) triples compute. For latency-sensitive flows (real-time chat), use lighter judge models (e.g., Haiku-class) for spot checks and reserve Sonnet/Opus-class judges for high-stakes operations.
- **Don't validate what you can compute.** If a field is "total = sum(line items)," check it mathematically. Don't route a math problem to a judge when a `sum()` call settles it. Save LLM validation for genuinely semantic checks — intent alignment, factual consistency, output plausibility.
- **The escalation path must actually exist.** Many teams implement a "surface to human" flag but never staff it. An unmonitored escalation queue is indistinguishable from a silent failure. If you add a HITL gate, instrument it so that human response SLA is tracked and alerted on.
