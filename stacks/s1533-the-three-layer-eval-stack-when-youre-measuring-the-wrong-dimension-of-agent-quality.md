# S-1533 · The Three-Layer Eval Stack — When You're Measuring the Wrong Dimension of Agent Quality

Your agent scores 91% on your eval suite. Your CI gate passes. Production incidents drop off the first week. Two months later, a security review finds your agent has been returning subtly hallucinated citations in 12% of sessions — your eval suite never caught it because it only scores the final answer, not the trajectory that produced it. The benchmark was real. The evaluation was incomplete.

## Forces

- **Most evals score outputs, not paths.** Task-success rate (did the agent finish?) and final-answer correctness (is the last message right?) are necessary metrics. But they miss whether the agent got there by the right route — using valid tools, avoiding loops, recovering from errors, and staying within policy. A correct answer from a wrong path is a liability in high-stakes domains. — [Braintrust, 2026](https://www.braintrust.dev/articles/ai-agent-evaluation-framework)
- **Vibe checks don't survive model drift.** Manually chatting with an agent to "see if it feels right" works for discovery, but LLMs are probabilistic: a prompt working at 99% today can drop to 92% within 24 hours due to model weight shifts or temperature changes. Prompt tweaks that pass a vibe check routinely regress in production. — [Google Cloud Blog, February 2026](https://cloud.google.com/blog/topics/developers-practitioners/from-vibe-checks-to-continuous-evaluation-engineering-reliable-ai-agents)
- **Trajectory errors compound silently.** In a 15-step agent run, a bad decision at step four corrupts steps five through fifteen. If your eval only checks the final answer, you miss the growing error surface. You also miss policy violations that occurred mid-trajectory but were later masked by fluent, confident output. — [MorphLLM, 2026](https://www.morphllm.com/ai-agent-evaluation)
- **The 95% failure rate is largely a measurement failure.** ~95% of enterprise AI projects fail — not primarily because the model is bad, but because organizations cannot measure whether the system is actually working in production. Defining success in a probabilistic environment and choosing the right evaluation tools remain the dominant unsolved challenges. — [Thoughtworks, June 2025](https://www.thoughtworks.com/en-au/insights/blog/machine-learning-and-ai/Evaluating-AI-agents-in-production)

## The move

Cover all three layers of agent evaluation with distinct tooling and signals, and run them continuously — not as a one-time gate but as a production infrastructure concern.

**Layer 1 — Final-answer scoring:** Score the last message against expected output or an accept-list. Use a curated golden set of 30–100 test cases with verified answers, versioned alongside code. Include hard edge cases and adversarial inputs, not just easy wins. This is your baseline contract. — [LLM Best Practices, May 2026](https://llmbestpractices.com/ai-agents/evaluation)

**Layer 2 — Trajectory scoring:** Score the sequence of steps: tool-call accuracy, path validity (no loops or redundant calls), step efficiency, and recovery behavior. Break tasks into milestones and score each. This layer catches the corrupted-path problem where a correct answer hides a broken process. — [Braintrust, 2026](https://www.braintrust.dev/articles/ai-agent-evaluation-framework); [AgentLens (arXiv 2607.06624), 2026](https://arxiv.org/abs/2607.06624v1)

**Layer 3 — Per-turn scoring in production:** Sample individual turns from live production traffic and score for policy adherence, jailbreak resilience, prompt leakage, and contextual appropriateness. This layer is invisible to offline evals and traces. Automated LLM-as-judge is the practical approach at scale, with strategic placement at three boundaries: before user-facing output, before irreversible tool execution, and on writes to persistent memory. — [Zylos Research, April 2026](https://zylos.ai/research/2026-04-10-llm-as-judge-production-agent-verification-2026); [MorphLLM, 2026](https://www.morphllm.com/ai-agent-evaluation)

**Run two eval modes in parallel:** Discovery mode uses exploratory testing to surface new failure patterns (human judgment, adversarial probes, red-teaming). Defense mode runs the regression suite to catch known failures before release. Discovery finds what you don't know; defense confirms what you already know doesn't regress. — [Google Cloud Blog, February 2026](https://cloud.google.com/blog/topics/developers-practitioners/from-vibe-checks-to-continuous-evaluation-engineering-reliable-ai-agents)

**Instrument CI/CD gates on the defense suite.** Every prompt change, model swap, or tool modification should run the full regression suite before merge. Score drift over time — if the defense suite score drops 2% month-over-month without intentional change, investigate. — [Braintrust, 2026](https://www.braintrust.dev/articles/ai-agent-evaluation-framework)

## Evidence

- **HN Ask HN (2025):** A practitioner building an LLM-powered product described the gap between eval tools and production needs: offline scorecards (benchmarks, rubrics, automated tests), production monitoring (drift, hallucination detection), ability to slice by model/prompt version and user segment, and CI/CD gating tied to product KPIs. No single tool covered all layers. — [Hacker News, 2025](https://hn.nuxt.dev/item/45644172)
- **Databricks (December 2024):** Enterprise teams using Agent Evaluation with synthetic data generation generated evaluation datasets (the GenAI equivalent of test suites) in minutes instead of weeks. One customer reported minimizing subject matter expert labeling time by pre-generating synthetic questions and answers, significantly accelerating their agent quality improvement cycle. — [Databricks Blog, December 2024](https://www.databricks.com/blog/streamline-ai-agent-evaluation-with-new-synthetic-data-capabilities)
- **AgentLens benchmark (arXiv 2026):** Standard code-agent benchmarks collapse evaluation to a single bit (did the task pass?). AgentLens instead evaluates the full trajectory — how the agent follows instructions, uses tools, verifies its own work, recovers from mistakes, and communicates — paired with formal verification for objective checks and LLM-written trajectory reviews for explainability. Used nightly in CI to catch product regressions across agent versions. — [arXiv:2607.06624, 2026](https://arxiv.org/abs/2607.06624v1)

## Gotchas

- **Golden sets go stale.** An eval set built for Q1 2025 may not reflect the distribution of inputs your agent sees in Q3 2025. Treat the golden set as a living artifact: retire obsolete cases, add cases from production failures, and track the match rate between eval inputs and production traffic distribution. — [LLM Best Practices, May 2026](https://llmbestpractices.com/ai-agents/evaluation)
- **LLM-as-judge is a second opinion, not ground truth.** Judges have their own failure modes: position bias (preferring first or last options), verbosity bias (preferring longer answers), and self-preference (favoring outputs similar to the judge's own style). Calibrate judges against human ratings before relying on them for release gates. Large proprietary judges (GPT-4o, Claude 3.7 Sonnet) offer better accuracy; small distilled judges (Patronus Lynx 8B, Prometheus 2 7B) offer 97% cost reduction at 0.88–0.95 accuracy for high-throughput inline checks. — [Zylos Research, April 2026](https://zylos.ai/research/2026-04-10-llm-as-judge-production-agent-verification-2026)
- **Intrinsic self-correction doesn't work without external grounding.** Telling an agent to "check your work" as a reasoning step consistently degrades performance on hard reasoning tasks. Self-correction only reliably helps when grounded in external feedback — unit test results, retrieval verification, or tool-output comparison. Don't confuse agent reflection with validated external checks. — [Zylos Research, April 2026](https://zylos.ai/research/2026-04-10-llm-as-judge-production-agent-verification-2026)
