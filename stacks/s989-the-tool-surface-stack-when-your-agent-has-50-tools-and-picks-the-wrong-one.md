# S-989 · The Tool Surface Stack — When Your Agent Has 50 Tools and Picks the Wrong One

You gave your agent twelve tools. It worked beautifully. Then someone added eight more for a new use case, and the agent started calling the Slack tool when it meant the database query — or worse, picked no tool at all and started reasoning from nothing. The tool count went up. The reliability went down. Nobody changed the model, the prompt, or the task. The only change was surface area.

## Forces

- **More tools = more capability, but more selection errors.** A model with 100+ tools doesn't just get better — it gets more opportunities to pick wrong, pass bad arguments, or fall back to reasoning without tool use at all.
- **MCP made the integration problem tractable, but not the selection problem.** Anthropic's Model Context Protocol (500+ community servers as of 2026) solved "how do I connect any model to any tool" — it did not solve "which tool should I use right now and with what parameters."
- **Security surfaces with surface area.** Destructive tools (database writes, Slack message sends, file deletions) need permission boundaries. An agent that can do anything is an agent that can cost you anything.
- **Token budgets make tool descriptions expensive.** Every tool's name, description, and parameter schema competes for context window space. With 50 tools, descriptions become terse; with 500, they're unmanageable.
- **Benchmarks measure tool use, not tool selection.** Tool calling benchmarks score whether the model invoked *a* tool correctly — not whether it invoked the *right* tool from the *right* set.

## The Move

Curate the tool surface, don't just grow it. The techniques that hold up in production:

- **Start with 3–5 tools that cover 80% of the use case.** One tool working at 90% reliability beats ten tools at 40%. Add tools only when a failure mode is genuinely uncovered by missing capability, not "future-proofing."
- **Namespace and group tools into capability clusters.** Rather than flat lists of 50 tools, organize into groups (e.g., `db_query_*`, `comm_*`, `code_*`) and expose groups dynamically based on task phase. A coding agent in write mode gets different tools than one in review mode.
- **Write tool descriptions as decision guidance, not feature lists.** "Read a file from the filesystem" is a description. "Use `read_file` when you need to inspect existing code or config before making changes; use `run_command` when you need to see what a command produces." — the second prevents misfires.
- **Use MCP as the tool protocol, not the tool repository.** MCP standardizes the interface (client/server, discovery, invocation). The 500+ servers in the community registry are starting points, not a shopping list. Pick the servers that match your security posture and eval results.
- **Instrument tool selection, not just tool execution.** Log which tool was selected, what parameters were passed, and whether the result was used. Tool selection errors (wrong tool, wrong params) are invisible if you only log execution failures.
- **Scope destructive tools behind confirmation.** Annotate destructive tools in MCP with read-only/destructive flags. Block's Goose, which runs company-wide at Block with thousands of daily users, uses MCP's permission annotations to require human confirmation for destructive actions — this is production-grade practice, not paranoia.

## Gotchas

- **Adding tools is easy. Removing them is political.** Once a tool is in production, teams build workflows around it. The cost of removing it exceeds the cost of keeping it. Gate additions with an eval requirement: no tool ships without a test showing it improves the success rate, not just adds coverage.
- **Tool descriptions compete with your context for space.** Parameter schemas, descriptions, and names are injected into the system prompt on every call. A 50-tool manifest at 200 tokens per tool × 20 agent steps = 200k tokens per session — in context you're paying for twice.
- **MCP servers vary wildly in maturity.** The 500+ community servers include proof-of-concept implementations, abandoned repos, and ones with known security issues. Stick to the official `modelcontextprotocol/servers` registry and verify active maintenance (commits in last 90 days, known security disclosures addressed).
- **Browser agents require sandboxing regardless of trust level.** Anthropic's own Computer Use documentation recommends dedicated VMs, allowlisted domains, no credential access, and human confirmation for financial or consent actions. "It's just a demo" is not a security posture.
- **Tool selection failure is silent.** A tool that executes successfully but was the wrong choice looks identical to a tool that was correct. You need structured logging of tool selection decisions — not just execution outcomes — to catch this class of failure.
