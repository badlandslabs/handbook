# S-1425 · The Browser Agent Stack — When Your Agent Needs to Click Things That Don't Have an API

Your agent can call APIs. It can write code. It can reason across documents. But the moment a vendor locks their workflow behind a web UI with no API access — a government portal, an internal tool, a partner's dashboard — your agent goes blind. You either hand-code a Playwright script that breaks the moment the DOM changes, or you accept that some workflows are simply unautomatable. Browser agents dissolve this boundary: the agent reasons about a page and drives it like a human, but at machine speed.

## Forces

- **Most enterprise software has no programmatic API.** Internal tools, legacy systems, and third-party SaaS products frequently expose their functionality exclusively through a browser UI. Browser agents give agents a universal interface to anything a human can reach with a mouse.
- **DOM fragility destroys scripted automation.** Playwright/Selenium scripts that target specific selectors or coordinates break constantly as web apps evolve. Browser agents reason about page content rather than targeting fixed DOM elements, making them self-healing when structure changes.
- **The write/read gap is massive.** State-of-the-art agents achieve >75% on read-heavy tasks (data extraction, comparison shopping) but only ~46% on write-heavy tasks (form filling, multi-step checkout, dynamic interactions). The technology is production-ready for extraction; write-heavy workflows still require human-in-the-loop for anything irreversible.
- **Infrastructure blocks agents more than intelligence.** CAPTCHA, proxy detection, login/session management, and rate limiting cause more production failures than model reasoning errors. Solving the intelligence problem doesn't help when a website blocks the request.
- **Browser agent traffic is becoming significant.** Agent-generated web traffic is projected to reach 25–35% of total web traffic by end of 2026, which will drive more sites to add friction or create official APIs in response.

## The move

**1. Start with the right tool for the job: browser agent for UI-only systems, MCP for everything with an API.** Browser agents are slower, more brittle, and more expensive than direct API calls. Use them only when no API exists. For everything else, MCP is the production-standard connection layer.

**2. Treat browser automation as a state-synchronization problem, not a click-targeting problem.** The dominant failure mode in browser agents isn't the model misunderstanding the page — it's the agent reasoning from a stale view of the page. After every action, capture the resulting state before the next reasoning cycle. The Agent Browser Protocol (ABP) project formalized this: freeze JavaScript execution and rendering after each action, then capture the resulting state, achieving 90.5% on Online Mind2Web (85.51% on hard tasks). Browser Use extracts all interactive elements with their content into a structured list presented to the LLM alongside the task — the LLM outputs `input_text(id 3, Hello World)` and Playwright executes the action.

**3. When the target app has an API, reverse-engineer it and wrap it in MCP instead of driving the UI.** Frigade's approach is instructive: run a browser inside the target app, observe the API calls the app makes to itself, and auto-generate reusable "recipes" (MCP-style tools) from those observations — including authentication, input/output schemas, and endpoint definitions. The agent uses the real API, not the UI, which is faster, more reliable, and doesn't break on DOM changes.

**4. Handle authentication carefully.** Browser agents frequently need to authenticate to the web apps they automate. Options include injecting session cookies, OAuth token management, or running authenticated browser sessions. Never expose credentials in plain text in prompts.

**5. Plan for infrastructure friction at scale.** Production browser agent deployments commonly hit: proxy blocks (solved by Browserbase or similar platforms that provide residential proxies), CAPTCHA (solved by CAPTCHA-solving services or simply routing around CAPTCHAs), rate limiting (solved by request throttling), and session timeouts mid-task (solved by session persistence and checkpoint/resume).

**6. Keep humans in the loop for write-heavy actions.** Anthropic's CUA (Computer-Using Agent) and OpenAI's Operator both require human confirmation for high-stakes actions in their default configurations. For internal automation where full autonomy is desired, implement a dry-run mode that logs intended actions before executing, and restrict full autonomy to low-stakes, reversible operations.

## Gotchas

- **Don't use browser agents for workflows that have APIs.** The moment you point a browser agent at something with a real API, you've chosen the slow, expensive, fragile path. Reverse-engineer the API and wrap it in MCP.
- **Don't assume an agent completed a task successfully just because it reached the end of its action sequence.** Agents report success when click coordinates resolve and the next screen loads — not when the intended outcome actually occurred. Build outcome verification steps, not just action completion checks.
- **Don't deploy browser agents against authenticated sessions without strict sandboxing.** Malicious web content can deliver prompt injection attacks through the agent to the authenticated session, leading to credential or session theft. Chrome DevTools Protocol access from an LLM is an uncontrolled surface area.
- **Don't skip infrastructure for production.** In benchmarks, infrastructure issues (proxies, CAPTCHA, auth, rate limits) cause more failures than model reasoning errors. Browserbase, Skyvern, and similar platforms exist because solving the intelligence problem doesn't help when a website blocks the IP.
- **Don't give browser agents full autonomy on irreversible actions.** Form submissions, payment flows, and account changes should require human confirmation in any production deployment. The 46% success rate on write-heavy tasks means roughly half of automated form submissions will fail or produce incorrect data — which is acceptable for a draft, not for a submitted tax filing.
