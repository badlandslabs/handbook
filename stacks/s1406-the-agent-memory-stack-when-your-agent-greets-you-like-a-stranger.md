# S-1406 · The Agent Memory Stack — When Your Agent Greets You Like a Stranger

Your agent worked fine for weeks. Then a long conversation triggered context compaction, a summary was written by a model with no skin in the game, and your agent greeted a six-month collaborator with "Hey! Great to meet you." This is not a session-boundary problem. This is a memory architecture problem. The agent didn't forget — it was made to forget by the system designed to help it remember.

## Forces

- **Context dilution, not session end** — the primary failure mode is not the session closing. It is silent context window eviction during long sessions: earlier decisions disappear not because the session ended, but because the model quietly overwrites them. Most teams don't discover this until a live failure
- **Memory is not a storage problem** — it is an ingestion, ranking, evaluation, and retrieval problem. The hard part is not where to put memories; it is whether the right memory comes back at the right time, in the right amount, under production latency constraints. A 6,000-entry memory corpus that returns noisy results is worse than no memory at all
- **The summarization trap** — automated context compaction summarizes what the algorithm thinks matters, not what actually matters. An agent-written self-summary at a natural breakpoint retains intent and emotional valence; a generic summarizer loses both
- **Vendor and machine lock-in** — Claude.md and rules files live in one tool on one machine. An agent that knows your project on laptop Cursor cannot carry that knowledge to desktop Claude Code. Cross-tool portability requires a storage layer outside the tool itself
- **Memory quality compounds** — a memory system with no quality gate accumulates noise indefinitely. In an ADD-only system, memories accumulate; in a DELETE system, the wrong deletion destroys context you can't recover. Both require active curation

## The move

Structure memory as a tiered retrieval system, not a flat store. Treat ingestion as a quality gate, not a pass-through.

- **Three-tier architecture (Core / Recall / Archival):** Core memory holds active constraints, identity, and goals — kept in context at all times. Recall memory is semantic: vector-stored, retrieved by query relevance with reranking. Archival memory is episodic and temporal, stored durably but accessed rarely. This mirrors how Mem0 and Letta structure their memory blocks — different tiers have different latency, cost, and retrieval characteristics
- **Ingestion quality gate, not pass-through:** Run every memory write through an extraction LLM call that produces structured facts, not raw text. Mem0's April 2026 algorithm (LoCoMo 92.5, LongMemEval 94.4) uses single-pass ADD-only extraction — one LLM call, no UPDATE/DELETE — so memories accumulate without overwrite. Agent-generated facts (confirmed actions) are stored as first-class facts alongside user statements
- **Sub-100ms retrieval for operational memory:** mem9 on TiDB Cloud targets sub-100ms retrieval for recall-layer queries. Redis Streams handles episodic event capture with temporal ordering. The memory tier closest to the agent loop must be latency-bounded — a 2-second retrieval round-trip inside an agent loop poisons the user experience
- **Server-centric memory logic:** mem9's architecture keeps integrations thin (OpenClaw, Claude Code, Cursor plugins) while the memory reasoning lives centrally. This means you improve memory behavior once, not in every plugin. Build the memory API as a service, not a library
- **Write self-summaries at natural breakpoints:** When the agent or user reaches a decision point, write a structured checkpoint to persistent storage. This checkpoint survives compaction. The author who had the "February 20th failure" fixed this by writing their own summaries proactively instead of waiting for automatic compaction
- **Keep critical constraints in system instructions, not context:** If a constraint must never be lost — "do not delete production data," "escalate before billing" — it lives in system instructions, not conversation context. Context is volatile. System instructions are not
- **Cross-tool portability via MCP:** The Model Context Protocol collapses N×M integrations to N+M for tool connections. For memory specifically, this means a memory MCP server (like Hmem's SQLite-backed hierarchical store or Mem0's cloud/self-host layer) gives any MCP-compatible agent access to the same persistent memory from any tool on any machine

## Gotchas

- **Compaction destroys nuance, not just volume:** Truncation drops not just old messages but the early context that gave later messages their meaning. "The project is stalled" means something different when you know the team had funding cut in week two. Automated summarizers strip this
- **Vector recall degrades as the corpus grows:** Semantic similarity search returns noise as the memory store scales. Early production systems saw "if recall becomes noisy as the memory corpus grows, trust disappears." Reranking (not just vector similarity) is required at scale
- **Memory without evaluation is a noise generator:** Every memory system that skips an ingestion quality gate accumulates facts that are wrong, outdated, or contradict other memories. Without a reconciliation step, the agent acts on contradictory ground truth
- **Memory corruption is silent and compounding:** A single bad memory entry — a misremembered constraint, a wrong tool name — can propagate across hundreds of agent actions. Unlike code bugs which surface immediately, memory corruption surfaces as slowly-degrading agent quality that teams often attribute to "model drift"
